# Issue #3 実装プラン — 診断結果グラフの種別とライブラリを選定する

## 0. このプランの性格

本プランは **実装コードを書くための設計書ではなく、「意思決定ドキュメント（spec）」を完成させるためのプラン** である。成果物は Markdown 1 ファイル（`docs/spec/result-dashboard.md`）、コード変更はゼロ。Issue #1（計算ロジック前提値）と Issue #2（入力フォーム UI）の確定結果を受け止め、`ResultDashboard` コンポーネント（マスター設計書 §3.2）の実装に必要な意思決定を固める。

実装ファイル（`src/components/ResultDashboard.tsx`、グラフ描画モジュール、アニメーション用フック等）は本 Issue のスコープ外。本仕様書を Issue #6（Next.js 雛形）・#8（グラフライブラリ追加）以降で参照して実装する。

---

## 1. 概要

### 何を決めるか
- 表示する **指標カードの構成と優先順位**（3 年止血／年間創出／3 年トータル、どれをヒーロー化するか、`insourcingGap` の扱い）。
- **グラフの種類**（円 / 棒 / 比較バー / 積み上げ / タイムライン / ウォーターフォール等）と点数（何個出すか）。
- **描画ライブラリ**の選定（Recharts / Chart.js / visx / 自前 SVG / D3）。
- **アニメーション方針**（数値カウンター実装方式、グラフ描画アニメーション、`prefers-reduced-motion` 配慮）。
- **レスポンシブレイアウト**（タブレット横持ち／スマホ縦／デスクトップ）。
- **警告バナーの UI 骨格**（`speedWarning = true` 時の配置・余白・色味。文言は Issue #4）。
- **PDF 出力（Issue #5）との連携方針**（同一グラフを html2canvas で画像化するか、PDF 用簡易版を別立てするか）。
- 表示単位・丸めルール（`calculation-logic.md §6` の「万円単位四捨五入」）をカウンター／軸／ツールチップすべてで統一する徹底方針。

### なぜ重要か（下流への影響）
- **Issue #5（PDF レポート）**: 本 Issue は #5 の直接の依存元。グラフが **SVG ベース** か **Canvas ベース** かで、html2canvas → jsPDF の経路での解像度・フォント崩れの発生有無が変わる。ここで決めた DOM/SVG 構造がそのまま PDF の画質を決める。
- **Issue #4（警告コピー）**: 警告バナーの **文言**は #4 スコープだが、ダッシュボード上の **配置（ヒーロー直下／最上部／カード群の間）** と **色トークン活用方針（Amber #F59E0B）** は #3 で確定しないと #4 がレイアウトを論じられない。
- **Issue #8（依存ライブラリ追加）**: 本 Issue で採用ライブラリが決まって初めて、#8 は `package.json` に追加するパッケージ名・バージョンを特定できる。
- **商談体験の核**: ダッシュボードは「診断する」ボタン押下後に経営者が最初に見る画面。最初の 1〜2 秒の視覚的インパクトと、そこから 30 秒以内に「自社にとって何が起きているか」を理解できる情報設計が、商談の勝ち筋を決める。

## 2. 意思決定ステップ（順序付き）

### Step 1. 表示指標の選び方と優先順位を決める

`calculation-logic.md §6.1` で「**3 年間の止血／年間の利益創出／3 年間のトータルインパクト**」の並列表示が規定済み。本 Step では「**並列の中でのヒエラルキー**」を確定する。

#### 選択肢A: 3 カード完全並列（フラット）
- 3 カードを同じサイズで横並び。どれも同じ視覚的重みで表示。
- **Pros**: 誠実でフラット。経営者が自分で重要な数字を選び取れる。
- **Cons**: 視覚的フックが弱く、「最初の 1 秒のインパクト」が出ない。商談ツールとしての訴求力が鈍る。

#### 選択肢B: トータルインパクトをヒーロー化（推奨）
- **3 年間のトータルインパクト**を画面上部に大きく配置（ヒーロー数値、カウンター付き）。その下に「3 年止血」「年間創出」を補助カード 2 枚で並列。
- **Pros**:
  - `calculation-logic.md §6.1` で「PDF サマリーは 3 年トータルインパクトを表紙訴求」と規定済みであり、**ダッシュボードと PDF の視覚的ヒエラルキーが一致**する。
  - 合算値がヒーローになることで「3 年でこれだけ価値が動く」という単一メッセージを最初の 1 秒で伝えられる。
  - 補助カードで内訳（止血・創出）を示すため、誠実性も担保。
- **Cons**: トータル値は合算で桁が大きくなりやすい。完全内製顧客（止血 = 0）でも「年間創出 × 3」がヒーローに出る点に注意（これは誠実な結果として許容する）。

#### 選択肢C: 3 年止血をヒーロー化
- 商品コンセプト「3 年間の止血」を反映して止血額を最大化表示。
- **Cons**: 完全内製顧客では止血 = 0 になり、ヒーロー数値が「0 万円」になる致命的な UX 破綻が発生。「誠実な結果」の原則に反してはいないが、ダッシュボードが「空白」に見える。

**推奨**: **選択肢B（トータルインパクトをヒーロー化）**。PDF との整合、完全内製顧客への耐性、商談ツールとしての訴求力の 3 点でバランスが最良。

#### 論点1-1. 「年間の利益創出」を「3 年間の利益創出」に置き換えるか
- `calculation-logic.md §6.1` は「**年間の利益創出**」と明記している。これを優先する。
- ただし補助カード内に「× 3 年 = ◯◯万円」の副次テキストを小さく添える（二段表示）ことで、合算への内訳理解を補強する。

#### 論点1-2. `insourcingGap` の可視化
- 係数値そのもの（例: `0.75`）をカード化すると経営者には理解しづらい。
- **採用方針**: 止血カードの直下に **「既に内製化されている ◯% を除外して試算」** という小さな注記テキストで透明性を担保する（数値としてのカード化はしない）。完全内製（`insourcingGap = 0`）の場合は「現状、理想形に近い運用のため削減余地は 0 万円」と表示切替。

#### 論点1-3. 「試算上の最大値」注記
- `calculation-logic.md §6.1` で規定。ヒーロー数値の直下に小さくフットノート形式で必ず入れる。

### Step 2. グラフの種類と点数を決める

指標カードで「絶対額」は伝わる。グラフは **「構成比」「時間変化」「対比」のいずれかの付加情報** を与えるものと位置付ける。

#### 候補と適性

| グラフ種別 | 適性 | 完全内製（止血=0）での挙動 | 備考 |
|---|---|---|---|
| 円グラフ（止血 vs 創出） | 構成比が一目瞭然 | 破綻（全体が創出 100% になる） | 2 要素の円グラフは Tufte 的に情報密度が低い |
| 積み上げ横棒（止血 + 創出） | 合算とその内訳を同時に見せられる | 止血セグメント消失、創出のみ表示でも破綻しない | トータル = ヒーローとの対応が取りやすい |
| 比較バー（現状支出 vs 導入後） | 「何が減るか」が直感的 | 意味を持たせづらい | 「現状支出」の定義が仕様にない |
| タイムライン（1 年目／2 年目／3 年目の累積） | 時間経過での効果蓄積が見える | 止血ゼロでも創出分が積み上がるので破綻しない | データは `threeYearSavings / 3` の按分になり、根拠薄 |
| ウォーターフォール（現状 → 止血 → 創出 → 合算） | ストーリー性が最も高い | 止血ステップが 0 でも自然に見える | 実装難度高、Recharts ネイティブにはなし（Composed で再現可） |

#### 推奨: **積み上げ横棒（止血 + 3 年創出）を 1 本、他はなし**
- 理由:
  1. **グラフは 1 つに絞る**（2 つ以上は商談テンポ劣化、画面占有過多）。
  2. 積み上げ横棒は「トータルインパクト = 止血 + 3 年創出」というヒーロー数値の内訳を視覚化できる。ヒーローと補助カードとグラフが同じ式を別表現で語る構造になり、情報の重複ではなく相互補強として機能する。
  3. 完全内製顧客でも破綻しない（止血セグメントが消え、創出セグメントだけの単色バーになる）。
  4. 軸を「万円」で統一でき、`calculation-logic.md §6` の表示ルールと完全整合。
  5. モバイル縦でも横棒なら崩れない（縦棒だと狭い画面で値ラベルが被る）。

#### 論点2-1. グラフ凡例と配色
- 止血 = Blue 500（#3B82F6、知的色）／創出 = Amber 500（#F59E0B、猫の温かみ）で、マスター設計書 §3.3 のアクセント 2 色をそのまま役割分担に使う。
- 凡例はグラフ直下に配置。色覚多様性対応として、凡例ラベルに小さなアイコン（Lucide `piggy-bank` / `sparkles` 等）を添える。

#### 論点2-2. ツールチップ
- 積み上げの各セグメントをホバー／タップすると、当該セグメントの金額を「万円」単位で表示。
- タブレット主用途のためホバーよりタップ優先。タップ後 3 秒で自動クローズ。

#### 論点2-3. 単一円グラフの可能性を棄却する理由
- 止血 vs 創出の 2 要素円は情報量が少なく、完全内製で破綻する。積み上げ棒の方が同じ情報をより堅牢に伝えられるため採用しない。

### Step 3. 描画ライブラリを選定する

#### 候補比較

| ライブラリ | 描画基盤 | バンドルサイズ(目安) | TypeScript | React 適合 | PDF 親和性 (html2canvas) | カスタマイズ自由度 | 日本語情報 |
|---|---|---|---|---|---|---|---|
| **Recharts** | SVG | 中（100kB 前後、tree-shake 可） | 公式型定義あり | React ネイティブ（宣言的JSX） | SVG のため拡大縮小に強く、html2canvas で劣化最小 | Tailwind でのラップ容易、shadcn/ui 周辺でも採用多 | 豊富 |
| Chart.js (+ react-chartjs-2) | Canvas | 中〜大 | `@types/chart.js` あり | ラッパ経由（やや間接的） | Canvas は html2canvas で **二重ラスタライズ**となり PDF 拡大時に劣化 | プラグインで拡張可だが Tailwind 非親和 | 豊富 |
| visx (Airbnb) | SVG | 小〜中（必要な部分だけ import） | 公式 TS | React ネイティブ | SVG で PDF 親和性高 | 最高（D3 ラッパ、自由度 = 記述量） | 少ない |
| 自前 SVG | SVG | 最小（0） | 自作 | React 直書き | PDF 親和性最高 | 無限 | N/A |
| D3 直接 | SVG / Canvas 両方 | 小（必要モジュール） | `@types/d3` | DOM を外側から操作、React と相性悪 | SVG 選択時は良好 | 最高 | 豊富 |

#### 評価軸ごとの重み付け
本プロジェクトで重要度が高い順に:
1. **PDF 出力との相性**（Issue #5 が直接依存） — SVG 必須級。
2. **実装速度**（フェーズ 1 で雛形着手までに決定済みである必要） — 宣言的 API の方が速い。
3. **TypeScript 型サポート** — プロジェクト方針（技術要件書 §2.1）。
4. **Tailwind / shadcn 周辺との親和性** — デザインガイドライン適用の容易性。
5. **バンドルサイズ** — Cloudflare Pages / LCP 1 秒目標との整合。
6. **カスタマイズ自由度（猫モチーフ・アクセント色対応）**。
7. **学習コスト・日本語情報量**。

#### 推奨: **Recharts**（本命）
- SVG ベースで html2canvas との相性が良い（PDF 生成時に再ラスタライズしても高解像度を保てる）。
- React 宣言的 API で `<BarChart><Bar /></BarChart>` の記述がそのまま JSX に収まり、Tailwind クラスや Lucide アイコンと自然に共存する。
- 公式 TypeScript 型定義があり、ESM / tree-shaking 対応済み。
- 積み上げ横棒は `<BarChart layout="vertical">` + `<Bar stackId="a" />` で素直に書ける。
- Next.js App Router ではクライアントコンポーネント境界（`"use client"`）内で使えば SSR 互換問題も軽微。

#### 対抗馬の扱い
- **Chart.js**: 採用しない。Canvas ベースは html2canvas で二重ラスタライズになり、PDF 拡大印刷時の劣化リスクが大きい。
- **自前 SVG**: 採用しない。グラフは 1 本だけでも、ツールチップ・アニメーション・凡例を自前で作るコストは仕様確定のスピードを毀損する。将来的にデザインが極端に凝る場合の選択肢として「未解決事項」に残す。
- **visx**: 対抗。Recharts より自由度は高いが、今回の要件（積み上げ棒 1 本）には過剰。Recharts で書けるものをわざわざ visx で書く理由はない。

#### バージョン指針（Issue #8 への申し送り）
- Recharts v2 系（React 18 対応、安定版）を採用する前提で仕様書に記載。ただし具体的なマイナー版固定は Issue #8 のスコープ。

### Step 4. アニメーション方針を決める

#### 論点4-1. 数値カウンター（0 → 最終値のカウントアップ）

| 選択肢 | 概要 | Pros | Cons |
|---|---|---|---|
| **A. 自前 `requestAnimationFrame` フック** | 50〜100 行の独自 `useCountUp` | 依存ゼロ、バンドル膨張なし、細かな制御（イージング、丸め単位）がしやすい | 初期実装の動作検証コストは自前持ち |
| B. Framer Motion の `animate()` / `motion.span` | `animate(0, target, { onUpdate })` で数値を補間 | API が洗練、他のアニメも一元管理できる | バンドル約 30〜50 kB、本件でカウンタ以外に使う用途がないと割高 |
| C. react-spring の `useSpring` | 物理ベース補間で自然な動き | 物理演出は魅力的 | 本件（整数カウントアップ）には過剰、バンドル増 |
| D. CSS transition | `content` プロパティの切替等で疑似的に | 最軽量 | 整数補間は CSS だけでは書けない（JS 補間が必要） |

**推奨**: **選択肢A（自前 `useCountUp` フック）**。
- カウンターは本アプリで唯一の数値アニメーション。Framer Motion を入れる理由がこれだけなら割に合わない。
- 「万円単位に四捨五入した整数」を補間する必要があり（`calculation-logic.md §6`）、イージングと最終値の丸め整合性を自分で持った方がむしろ安全。
- duration は **1,200 ms、イージング `easeOutCubic`**。商談で「0 → 3,600 万円」へ 1.2 秒かけてカウントアップする演出が「インパクトを出しつつ焦らせない」値として推奨。

#### 論点4-2. グラフ自体のアニメーション
- Recharts は標準でバーの伸長アニメーションを持つ（`isAnimationActive` / `animationDuration`）。
- **推奨**: **有効化、duration 800 ms、カウンターと同時開始**。バーが伸びる動きとカウンターが走る動きが揃うことで、視覚演出として一体感が出る。

#### 論点4-3. `prefers-reduced-motion` 対応
- タブレット商談では視覚演出が商談の価値の一部であり、原則アニメーション有効。
- 一方で **アクセシビリティ遵守** の観点から、`prefers-reduced-motion: reduce` を尊重した場合は：
  - カウンターは即時最終値表示（duration 0）。
  - グラフのバー伸長もスキップ（`isAnimationActive={false}`）。
- **推奨**: **`prefers-reduced-motion` を尊重する**。商談用 iPad で OS 設定を「視差を減らす」にしている顧客はまれだが、Web アプリとしての基本マナーとして尊重する。

#### 論点4-4. アニメーションの発動タイミング
- マウント時に一度だけ発動（ダッシュボード初回表示時）。
- 確認画面から「修正」で戻って再度「診断する」を押した場合は **再発動**（新しい数値に対して再びカウントアップ）。

### Step 5. レスポンシブレイアウトを決める

#### 5-1. ブレークポイント
`input-form.md §9` と整合させる。Tailwind 既定のブレークポイントに従う。

| ブレークポイント | 主用途 | 優先度 |
|---|---|---|
| `< 640px` (sm 未満) | スマホ縦 | 次点（商談の代替デバイス） |
| `640px – 1023px` (sm / md) | **タブレット（横・縦持ち商談）** | **最優先** |
| `≥ 1024px` (lg 以上) | デスクトップ（営業担当の下準備・社内プレゼン） | 副次 |

#### 5-2. レイアウトの基本パターン

```
┌──────────────────────────────────────────────┐
│ [警告バナー (speedWarning=true 時のみ)]        │
├──────────────────────────────────────────────┤
│         ヒーロー: 3 年トータルインパクト        │
│         ◯◯◯◯ 万円  （カウンター）           │
│         ※試算上の最大値                      │
├──────────────────────────────────────────────┤
│  [止血カード]          [年間創出カード]        │
│  ◯◯万円               ◯◯万円 (×3=◯◯)       │
│  注記: 内製化 n%除外                          │
├──────────────────────────────────────────────┤
│         積み上げ横棒グラフ (1 本)              │
│         Blue=止血 / Amber=3年創出             │
├──────────────────────────────────────────────┤
│         [PDF ダウンロード] [再診断]           │
└──────────────────────────────────────────────┘
```

#### 5-3. 各ブレークポイントでの挙動

| 要素 | タブレット横 (≥768px) | スマホ縦 (<640px) | デスクトップ (≥1024px) |
|---|---|---|---|
| ヒーロー | 中央揃え、フォント clamp(2.5rem, 6vw, 4rem) | 同上、フォント小さめ | 最大幅 720px でセンタリング |
| 補助カード | 2 カード横並び (grid-cols-2) | 縦スタック (grid-cols-1) | 2 カード横並び、カード間余白 24px |
| グラフ | 横幅 100%（最大 960px） | 横幅 100%、高さ 200px | 最大 960px でセンタリング |
| 警告バナー | 全幅、高さ 64px | 全幅、高さ 72px（改行分） | 全幅、最大幅 960px |
| ダッシュボード最大幅 | 1024px | 制約なし（`px-4` パディング） | 1024px |

#### 5-4. 1 画面収まり保証（タブレット横）
- タブレット横持ち（iPad 9th 以降の 810px × 1080px、横持ち 1080 × 810）を基準に、**警告バナー + ヒーロー + 補助カード 2 枚 + グラフ + CTA** が スクロールなしで収まるよう設計。
- ヒーロー数値の `font-size` は最大 `clamp(2.5rem, 6vw, 4rem)`、縦方向のセクション間スペースは `space-y-6` (24px) を基調とする。

### Step 6. 警告バナー（Issue #4 連携）の UI 骨格を決める

#### 6-1. 位置
- **ダッシュボード最上部（ヒーローの直上）**。
- 理由: `speedWarning = true` は「今の 3 ヶ月以上の更新待ちで CRITICAL OPPORTUNITY LOSS が発生」という緊急シグナル。数値カードを見る前に認識してもらう必要がある。
- `speedWarning = false` 時はバナーを出さず、ヒーローが最上段に繰り上がる。

#### 6-2. 色・アイコン
- 背景: Amber 500（#F59E0B）の透明度 10〜15%（`bg-amber-500/10` 相当）。枠線: Amber 500 実線。
- アイコン: Lucide `alert-triangle`、色は Amber 600。
- 文言: **Issue #4 のスコープ**。本仕様書ではプレースホルダ `"{{CRITICAL_OPPORTUNITY_LOSS_MESSAGE}}"` として記載し、#4 完了後に差し替える運用とする。

#### 6-3. 余白・高さ
- タブレット: 高さ 64px、水平パディング 24px、アイコンと文言の間 12px。
- スマホ: 高さ 72px（2 行改行前提）、水平パディング 16px。

#### 6-4. アニメーション
- マウント時にフェードイン 300ms のみ。カウンターと同時でよい（目立たせすぎると他の要素を邪魔する）。

### Step 7. 表示単位と丸めルールの徹底方針

`calculation-logic.md §6` の「**万円単位で四捨五入、UI に『万円』単位を付与、ゼロ値はそのまま表示**」を以下すべてで統一する:

### Step 8. PDF 向けスナップショットの分岐

Issue #5 は #3 に依存するため、本 Issue で **「PDF が何をスクショするのか」の方針**を定めておく必要がある。

#### 選択肢A: 画面と同一コンポーネントをそのまま html2canvas
- **Pros**: コンポーネント二重化なし、常に画面と PDF が一致。
- **Cons**:
  - インタラクティブ要素（ツールチップ、アニメーション、ボタン）がそのまま画像化されうる。PDF 生成直前に「アニメーション完了状態」「ツールチップ非表示」に揃える工夫が必要。
  - レスポンシブレイアウトがそのまま出るため、A4 比率と合わない可能性。

#### 選択肢B: PDF 専用の「スナップショット版コンポーネント」を別立て（推奨）
- 隠し DOM（`position: absolute; left: -9999px;` 等）に **A4 比率固定・アニメーション無効・インタラクション無効**のダッシュボード簡易版を描画し、html2canvas でそれを画像化。
- マスター設計書 §3.2 の `PdfGenerator` が「**隠しDOM要素（A4比率）を画像化しPDFに変換**」と明記しており、この方針と完全整合。
- **Pros**: PDF レイアウト最適化と画面インタラクションを分離できる。画面側の変更で PDF が壊れない。
- **Cons**: コンポーネントが二重化する。ただし `ResultDashboard` を `props` で受ける形にリファクタしておけば、PDF 版は「同じ props を受けて layout だけ変える」薄いラッパで済む。

#### 推奨: **選択肢B（PDF 用スナップショット版を別立て）**。
- 本仕様書では「**ダッシュボードを実装する際、見た目の責務を持つ描画層 `DashboardView`（props ベース、副作用なし）と、データ取得・アニメーション等の副作用を担うコンテナ `ResultDashboard` を分離する**」設計指針のみ記載。`DashboardView` を PDF 用コンポーネントが別レイアウトで再利用する想定。詳細レイアウトは Issue #5 のスコープ。

## 4. 考慮事項・リスク

### R1: Recharts と Next.js App Router の SSR 境界
- Recharts は `window` / `document` を前提としたコードを含むため、App Router でのサーバーコンポーネント境界で `"use client"` 指定が必要。
- → 本仕様書に「**`ResultDashboard` および `DashboardView` はクライアントコンポーネント**」と明記。Issue #6 以降の実装で `"use client"` を冒頭に置く。

### R2: html2canvas と SVG グラフの互換性
- html2canvas は SVG をラスタライズできるが、バージョンによっては `foreignObject` や外部フォントの扱いに差が出る。
- → PDF 用スナップショット版（§8）では **system フォントスタックまたは next/font のローカル埋め込みフォント**を使用し、外部 CSS 依存を作らない方針を仕様書に明記。Issue #5 実装時の検証項目として申し送り。

### R3: カウンター補間と万円丸めの不整合
- 補間途中値 `2,345,678` 円を毎フレーム `Math.round(/10000)` すると、末尾が飛び飛びに変化（234 → 234 → 235…）。これは意図通りだが、最終値 `3,600,000` → 表示 360 で止まることを担保する必要がある。
- → 仕様書に「**補間完了時、`rAF` の最終コールバックで target 値をそのまま `formatManYen(target)` で表示する**」と明記。浮動小数誤差対策。

### R4: 完全内製顧客（止血 = 0）でのグラフ破綻
- 積み上げ棒の止血セグメントが 0 の場合、Recharts は自然に描画しない（単色バーになる）。破綻はないが、凡例に「3 年間の止血（0 万円）」と出続けるのは気になる。
- → 仕様書に「**`threeYearSavings === 0` のときは凡例から止血項目を動的に除外する**」と記載。または凡例を維持しつつカード側注記で「現状、理想形に近い運用のため削減余地は 0 万円」を表示。推奨は後者（凡例は構造を保ち、文脈は注記で語る）。

### R5: ヒーロー数値の桁爆発（大企業顧客）
- `monthlyVendorCost` 上限 1 億円/月 × 36 ヶ月 + 創出 = 数十億円級のヒーロー数値が発生しうる。`clamp(2.5rem, 6vw, 4rem)` では桁溢れはしないが、読みにくくなる。
- → 仕様書に「**ヒーロー数値が 10 億円（= 10 万万円）を超える場合、`formatManYen` の代替として `formatManYen_compact` で『◯億◯万円』表示に切り替える**」を記載。この分岐のための閾値と表記ルールを別途定義。

### R6: 警告バナー文言の長さ不確定（Issue #4 待ち）
- 文言が 1 行に収まるか 2 行になるかでバナー高さが変わり、タブレット横 1 画面収まり保証（§5-4）に影響する可能性。
- → 仕様書に「**バナー高さは最小 64px・最大 96px を許容、超える場合はヒーロー数値の `font-size` を `clamp` の下限側に寄せて吸収**」と記載。Issue #4 完了後のレビューで最終確認。

### R7: アニメーションの再発動制御
- 確認画面 → 修正 → 再診断 の動線で、同一コンポーネントのリマウントが起こらない実装パターンだとカウンターが再発動しない。
- → 仕様書に「**`key` prop に `result` オブジェクトのハッシュを渡してリマウントを強制する**」設計指針を記載。または `useEffect` で `result` 依存にしてカウンター state をリセット。

### R8: Recharts バンドルサイズ
- Recharts は全体で 100kB 前後だが、tree-shaking が効きにくい箇所がある。Cloudflare Pages LCP 1 秒目標への影響は要計測。
- → 仕様書に「**初回表示には Recharts は不要（入力フォーム画面のみ）のため、`ResultDashboard` は `next/dynamic` で遅延ロードする**」指針を記載。Issue #6 以降の実装で適用。

### R9: `insourcingGap` 注記の表記
- 「内製化 75% 除外」と書くと経営者は「75% 割引されている」と誤読しうる。正しくは「内製化 25% 相当分を削減余地から除外」。
- → 仕様書に「**注記は『既に内製化されている ◯% 相当分は削減余地から除外済み』と明示する**」文言パターンを記載。`◯` 部分は `insourcingLevel * 100`（= 除外率）で展開。Issue #4 のカテゴリ別表示文（`calculation-logic.md §4.4`）とトーンを揃える。

### R10: PDF 版での prefers-reduced-motion
- PDF 生成時、ユーザーの OS 設定に関わらず **必ずアニメーション無効のスナップショットを撮る必要がある**。`animated={false}` を props で強制する。
- → 仕様書の `DashboardViewProps` に `animated` を必須級の意味付けで含める。
