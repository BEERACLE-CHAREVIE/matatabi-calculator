# Issue #1 実装プラン — 計算ロジックの前提値を確定する

## 0. このプランの性格

本プランは **実装コードを書くための設計書ではなく、「意思決定ドキュメント（spec）」を完成させるためのプラン** です。成果物は Markdown 1〜2ファイル、コード変更はゼロです。Issue #1 はフェーズ0の根幹であり、#2〜#5 の全てがここに依存します。

---

## 1. 概要

### 何を決めるか
- ROI 計算式で使用される **固定値4種（時給2,500円／1日2時間／月20日／年3回改修／36ヶ月）** の扱いを「固定定数」「可変入力」「ハイブリッド」のいずれにするかを確定する。
- 入力フォームの「**内製化状況**」について、選択肢セットと、計算式に対する反映方法（係数 or 表示のみ）を確定する。
- 上記を反映した **最終計算式をTypeScript擬似コードレベルで明文化** する。

### なぜ重要か（下流への影響）
- **#2（入力フォームUI）**: 「固定値を可変にする」と決めた瞬間、フォーム項目数が5項目から増え、UIの複雑度が変わる。
- **#3（グラフ）**: 計算式から出る出力項目（削減額・創出額・機会損失額など）の数と性質がグラフ種別を決める。
- **#4（警告コピー）**: 「内製化状況」を係数化するなら、低内製化の顧客に対して特定の警告文を出す設計になりうる。
- **#5（PDF）**: PDFの記載項目と、「前提値」セクションの有無が変わる。
- **説明責任**: 経営者に数値の根拠を問われた時、即答できる前提値設計でなければ営業ツールとして機能しない。

## 2. 意思決定ステップ（順序付き）

### Step 1. 固定値の扱いを決める（最重要・最優先）

対象となる5つの定数:

| 変数 | 現行値 | 単位 |
|---|---|---|
| 時給単価 | 2,500 | 円/時 |
| 1日あたり作業時間 | 2 | 時間 |
| 月あたり稼働日数 | 20 | 日 |
| 年あたり改修回数 | 3 | 回 |
| 試算期間 | 36 | ヶ月 |

**3つの選択肢**:

#### 選択肢A: 完全固定定数
- **実装**: `src/lib/constants.ts` にすべて定数として埋め込む。ユーザーは一切変更不可。
- **Pros**:
  - 実装が最速・最小。フォームは5項目のまま。
  - 営業トークで「業界標準値で試算」と一言で済み、説明責任の境界が明確。
  - 商談ごとに結果がぶれず、資料として標準化しやすい。
- **Cons**:
  - 「うちは時給3,500円だ」「月22日稼働だ」と言われた時に即応できない。
  - 業種・規模が大きく異なる顧客に対して説得力を欠く。

#### 選択肢B: 全て可変パラメータ
- **実装**: 各定数を入力フォームに昇格。フォームが10項目以上になる。
- **Pros**:
  - 顧客個別の前提に完全フィットできる。
  - 数値根拠の説明責任を顧客自身に委ねられる（「お客様の入力値で計算しました」）。
- **Cons**:
  - フォームが重くなり、商談の場でのテンポが悪化（タブレットで10項目は苦痛）。
  - 入力のばらつきで営業担当が結果をコントロールしづらい。
  - 経営者に「そもそも時給いくらで計算すべきか」を尋ねる時点で商談のイニシアチブを失う。

#### 選択肢C: ハイブリッド（推奨）
- **実装**: 定数として「デフォルト値」を持ちつつ、「詳細設定」トグルで上書き可能にする。デフォルトでは閉じた状態。
- **Pros**:
  - 通常商談では5項目のシンプルフォームのまま（A と同じUX）。
  - 「前提値の根拠は？」と問われた時、折り畳みを開いて「時給2,500円（業界平均値）で試算、ご希望なら変更できます」と説明できる。
  - 経営者視点でも透明性が担保される（ブラックボックスでない）。
- **Cons**:
  - UI実装が増える（折り畳みコンポーネント、上書きロジック）。ただし軽微。
  - 「デフォルト値の根拠」はどのみち明文化が必要（→ Step 4 で対応）。

**推奨**: **選択肢C（ハイブリッド）**。商談テンポを損なわず、かつ説明責任に耐える。ただし、**試算期間「36ヶ月＝3年」と「年3回改修」は商品コンセプト（「3年間の止血」）に結合しているため固定**として扱い、可変にするのは「時給」「1日作業時間」「月稼働日数」の3つに限定する。

### Step 2. 「内製化状況」の選択肢を決める

**3つの選択肢**:

#### 選択肢α: 段階セレクト（5段階、推奨）
```
0%（完全ベンダー依存）
~25%（一部内製）
~50%（半分内製）
~75%（大半内製）
100%（完全内製）
```
- ラジオボタンまたはセグメントコントロール。

#### 選択肢β: 2択（内製 / 非内製）
- シンプルだが、中間状態の顧客の実態を捉えられない。

#### 選択肢γ: スライダー（0〜100%）
- 数値入力の自由度は高いが、「うちは何%なのか」を経営者が即答できない。

**推奨**: **選択肢α（5段階）**。経営者の直感で選べ、離散値なので計算・表示も安定する。

---

### Step 3. 「内製化状況」の計算式への反映方法を決める

#### 選択肢i: 表示のみ（計算には使わない）
- 入力値はダッシュボードの「現状サマリー」に表示するだけ。
- **Pros**: 説明責任が軽い。計算式がシンプル。
- **Cons**: 入力させた意味が薄く、ユーザーから「なぜ聞いた？」と感じられる。

#### 選択肢ii: 係数として計算式に乗算（推奨）
- 「内製化率が低いほど削減余地が大きい」というストーリーと整合。
- 具体的には **「3年間の止血」に内製化ギャップ係数を乗算**:
  ```
  内製化ギャップ係数 = (1 - 現状内製化率)
  例: 0% → 1.0（満額削減可）、25% → 0.75、50% → 0.5、75% → 0.25、100% → 0.0
  ```
- **Pros**: 入力が結果に反映されるのでユーザー体験が誠実。完全内製の会社は「削減余地ゼロ」と正直に出る。
- **Cons**: 係数の根拠を問われる。→ 「既に内製化されている割合分は既に削減済みなので、残りのベンダー委託部分のみが削減対象」というロジックで説明可能。

#### 選択肢iii: カテゴリ分岐（警告メッセージのみ切替）
- 計算式には入れず、「内製化0%」の顧客には「伸びしろ大」、「100%」の顧客には「維持フェーズ」など表示文を切り替える。

**推奨**: **選択肢ii（係数化）＋ 選択肢iii の併用**。計算に反映しつつ、カテゴリごとの訴求文も出し分ける。

---

### Step 4. デフォルト値の根拠を明文化

各デフォルト値に「経営者に説明できる1行の根拠」をセットで記録する。例:

| 値 | 根拠（例） |
|---|---|
| 時給2,500円 | 中小企業の平均的なホワイトカラー人件費（社会保険料等込みの実質コスト）を参照 |
| 1日2時間 | 定型業務・手作業の1人あたり平均所要時間の業界アンケート等を参照 |
| 月20日 | 標準的な営業日数 |
| 年3回改修 | Web/システムの典型的な改修頻度（四半期ごとを想定し、うち1回を軽微として除外） |
| 36ヶ月 | 商品コンセプト「3年間の投資対効果」に準拠 |

※ 実際の出典は仕様書作成時に Issue #1 の担当者が決定する。現時点ではプレースホルダで可。

### Step 5. 最終計算式を擬似コードで明文化

以下を仕様書に記載する:

```ts
// ───────────────── 定数（固定） ─────────────────
const MONTHS_IN_PERIOD = 36;        // 3年
const REPAIRS_PER_YEAR = 3;          // 年3回
const YEARS_IN_PERIOD = 3;

// ───────────────── デフォルト値（ハイブリッドで上書き可） ─────────────────
const DEFAULT_HOURLY_WAGE = 2500;    // 円/時
const DEFAULT_HOURS_PER_DAY = 2;     // 時間/日
const DEFAULT_DAYS_PER_MONTH = 20;   // 日/月
const MONTHS_PER_YEAR = 12;

// ───────────────── ユーザー入力 ─────────────────
type InsourcingLevel = 0 | 0.25 | 0.5 | 0.75 | 1.0;

interface Inputs {
  monthlyVendorCost: number;           // 月額ベンダー費用（円）
  repairCost: number;                  // 単発改修費用（円）
  manualWorkerCount: number;           // 手作業人数
  updateWaitMonths: number;            // 更新待ち期間（月）
  insourcingLevel: InsourcingLevel;    // 内製化状況
  // ↓ 詳細設定（オプション。未指定時はデフォルト値）
  hourlyWage?: number;
  hoursPerDay?: number;
  daysPerMonth?: number;
}

// ───────────────── 計算式 ─────────────────
function calculate(input: Inputs) {
  const hourlyWage = input.hourlyWage ?? DEFAULT_HOURLY_WAGE;
  const hoursPerDay = input.hoursPerDay ?? DEFAULT_HOURS_PER_DAY;
  const daysPerMonth = input.daysPerMonth ?? DEFAULT_DAYS_PER_MONTH;

  // 内製化ギャップ係数（既に内製化された分は削減余地から除外）
  const insourcingGap = 1 - input.insourcingLevel;

  // 3年間の止血（削減額）
  const rawSavings =
    input.monthlyVendorCost * MONTHS_IN_PERIOD +
    input.repairCost * REPAIRS_PER_YEAR * YEARS_IN_PERIOD;
  const threeYearSavings = rawSavings * insourcingGap;

  // 年間の利益創出額
  const annualProfitCreation =
    input.manualWorkerCount *
    hoursPerDay *
    daysPerMonth *
    MONTHS_PER_YEAR *
    hourlyWage;

  // スピード警告
  const speedWarning = input.updateWaitMonths >= 3;  // 3ヶ月以上で発動

  return {
    threeYearSavings,
    annualProfitCreation,
    threeYearProfitCreation: annualProfitCreation * YEARS_IN_PERIOD,
    totalThreeYearImpact: threeYearSavings + annualProfitCreation * YEARS_IN_PERIOD,
    speedWarning,
    insourcingGap,
  };
}
```

**確定事項（この擬似コードが仕様書の中心になる）**:
- 「スピード警告」は `updateWaitMonths >= 3` で発動（ドキュメントの「3ヶ月以上」を厳密化）。
- 「3年間のトータルインパクト」を合算指標として追加定義するかは Step 6 で判断。
- 内製化係数は「止血」にのみ乗算。「創出」には乗算しない（AI自動化による創出はベンダー内製化と独立のため）。

---

### Step 6. 付加指標と丸めルールを決める

- 丸め: 円単位か万円単位か。経営者向けなら **万円単位での表示**（内部計算は円単位で保持、UI表示時に丸める）。
- 合算指標「3年トータルインパクト」を出すか。→ **出す**（PDFの表紙・サマリーで訴求力が高い）。
- 年間創出額を3年分に展開するか。→ **する**（「3年間の創出」= 年間創出 × 3）。

## 4. 考慮事項・リスク

### R1: 固定値を可変にするとUIが複雑化
- ハイブリッド案採用時、「詳細設定」折り畳みのデフォルト状態・ラベル文言・モバイル表示が新たな設計論点になる。→ Issue #2 のスコープで吸収される前提でOK。

### R2: 「内製化状況」係数化の説明責任
- 「なぜ内製化率を乗算するのか」の一行説明を仕様書に明記する。ロジックは **「既に内製化された分は既に自社内にコストが吸収済みなので、ベンダー流出コストからは除外」**。
- 完全内製（100%）の顧客は「削減額ゼロ」と出る。これは誠実な結果だが、営業トークとしては「既に理想形、維持フェーズですね」に切り替える必要あり → Issue #4（警告コピー）で吸収。

### R3: 業種・規模による前提差
- 例: 製造業の時給とIT企業の時給は大きく異なる。
- → ハイブリッド案の「詳細設定」で吸収。業種別プリセットを複数用意するのは **今回スコープ外**（将来拡張として仕様書の「8. 未解決事項」に記録）。

### R4: 前提値の出典不足
- 現状、2,500円 / 2時間 / 20日などの出典は設計ドキュメントに明記されていない。Issue #1 の完了条件として「各デフォルト値に少なくとも1行の根拠コメントを付ける」を必須化する。厳密な統計出典までは求めず、「商談で聞かれた時に答えられる説明ロジック」のレベルで可。

### R5: 経営者視点での数値の「盛り」問題
- 係数なし計算だと「うちのシステムは月額100万円→3年で3,600万円削減！」と過大に見える恐れ。
- → 内製化係数の導入により、自然にキャリブレーションされる。加えて、UI表示で「試算上の最大値」であることを注記する方針を仕様書に記載。

### R6: マスター設計書と仕様書の二重管理
- `.claude/` のマスター設計書と `docs/spec/calculation-logic.md` で情報が二重化する。
- → マスター設計書の §3.1 に「確定仕様は `docs/spec/calculation-logic.md` を参照」と1行の追記を推奨（ただし本Issueで .claude/ を編集するかは別途判断）。
