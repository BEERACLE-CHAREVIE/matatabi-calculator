# Issue #2 実装プラン — 入力フォーム5項目の UI 仕様を確定する

## 0. このプランの性格

本プランは **実装コードを書くための設計書ではなく、「意思決定ドキュメント（spec）」を完成させるためのプラン** です。成果物は Markdown 1ファイル（`docs/spec/input-form.md`）、コード変更はゼロ。Issue #1（計算ロジック前提値）の確定結果を入力側から受け止め、Issue #6（Next.js 雛形作成）以降の `InputForm` 実装がそのまま着手できる状態にまで「入力 UI の意思決定」を固めることが目的です。

実装ファイル（`src/components/InputForm.tsx`、`src/lib/validation.ts` 等）は本 Issue のスコープ外。本仕様書を Issue #6 以降で参照して実装します。

---

## 1. 概要

### 何を決めるか
- 基本 5 項目（月額ベンダー費用／改修費用／手作業人数／更新待ち期間／内製化状況）それぞれの **UI 種別**・**単位**・**バリデーション範囲（最小値／最大値／整数制約／刻み）**・**必須/任意**・**プレースホルダ/ヘルプ文言の戦略** を確定する。
- 詳細設定 3 項目（時給／1日作業時間／月稼働日数）の UI 詳細（折り畳み内の配置・ラベル・範囲）を確定する。
- **Step-by-Step UI の区切り方**（1項目1ステップ／複数項目1画面／単一画面スクロール）を確定する。
- バリデーションの実装方針（**Zod 等ライブラリ導入の是非**、エラー表示タイミング、空値・ゼロ値の扱い）を確定する。
- **タブレット／スマホ最適化**（数値キーボード呼び出し、タップ領域、入力インライン表示）の方針を確定する。

### なぜ重要か（下流への影響）
- **#3（グラフ・ダッシュボード）**: 入力単位（円 vs 万円）が結果表示単位と整合しないと、ユーザーが「入力した金額と結果のスケールが合わない」と混乱する。
- **#4（警告コピー）**: 更新待ち期間の UI 種別が「3ヶ月以上で警告」の発動タイミングに直結。連続値（スライダー）と離散値（セレクト）で警告の出方が変わる。
- **#5（PDF）**: 入力サマリーを PDF に載せる場合、「入力された値の表示形式」が PDF レイアウトの前提になる。
- **#6（Next.js 雛形）**: `Inputs` 型に対するバリデーションスキーマの形式（Zod / 手書き）を決めておかないと実装者が迷う。
- **商談体験の核**: 入力フォームは「商談中に経営者の目の前でタブレットを操作して数値を埋めていく」場面そのもの。ここが重ければアプリ全体の価値が崩壊する。**商談テンポ・迷わない UI・経営者が自分で入力できる** ことが最優先設計原則。

## 2. 意思決定ステップ（順序付き）

### Step 1. Step-by-Step UI の区切り方を決める（UX フレームを最初に確定）

他項目の UI 選択がこのフレームに依存するため最初に決める。

#### 選択肢A: 1項目 1ステップ（完全ウィザード方式）
- 画面に 1 つの質問のみを大きく表示、「次へ」ボタンで進む。合計 5 ステップ（+ 詳細設定）。
- **Pros**:
  - 経営者が「今この 1 問に集中すればいい」と迷わない。
  - タブレット商談で 1 項目がフルスクリーン級に大きく出るので、目の前のお客様にも見せやすい。
  - 進捗インジケータ（「2/5」）が出せて、終わりが見えるのでストレスが少ない。
- **Cons**:
  - 入力の前後文脈（「さっき何を入れたか」）が見えにくい。
  - 「最後にまとめて確認」画面が別途必要になる。
  - 画面遷移が 5 回発生し、全体所要時間は体感で長くなりうる。

#### 選択肢B: 複数項目を 1 画面（2〜3 ステップ構成）
- 例: ステップ1「コスト関連（月額・改修）」→ ステップ2「体制関連（人数・待ち期間・内製化）」→ 確認。
- **Pros**:
  - 関連項目がグループで見えて意味的に理解しやすい。
  - 画面遷移が少なく、商談テンポが良い。
- **Cons**:
  - グループ化の切り方の良し悪しで UX 品質が揺れる。
  - 1 画面の情報量がやや増え、モバイル縦画面では圧迫感が出やすい。

#### 選択肢C: 単一画面スクロール（Step-by-Step ではない）
- 5 項目 + 詳細設定を縦並び、末尾に「診断する」ボタン。
- **Pros**:
  - 実装が最も軽い。全体像が常に見える。
  - 入力修正（前に戻る）が自然。
- **Cons**:
  - マスター設計書 §3.2 で「Step-by-Step 形式」と明記されており、**この選択肢を取るならマスター設計書側の更新が必要**。
  - 商談中、経営者が一覧で見て「多いな」と感じる心理的ハードルが出る。

**推奨**: **選択肢A（1項目1ステップ）＋ 末尾に確認画面**。マスター設計書の方針にも整合し、商談での「1問ずつお答えいただく」会話の流れにも最も適合する。内製化状況は選択肢型で一瞬で終わるため、5 ステップでも体感は軽い。ただし「戻る」ボタンは各ステップで必須（誤入力時のストレス回避）。

### Step 2. 月額ベンダー費用の UI を決める

#### 論点2-1. UI 種別
| 選択肢 | Pros | Cons |
|---|---|---|
| 数値入力（テキストフィールド） | 正確な金額を入れられる。桁数に制約がない。 | タブレットでキーボードを開く必要。入力ミスが発生しうる。 |
| レンジスライダー | タップだけで完了、商談テンポ最速。 | 細かい金額（例: 87 万円）を入れづらい。 |
| 数値入力 + ±ボタン（ステッパー） | タップとキーボードの中間案。 | 大きな桁数の金額では操作回数が増える。 |

**推奨**: **数値入力（モバイルでは `inputmode="numeric"` で数値キーボード強制）**。金額は正確性が必要で、スライダーでは説明責任が揺らぐ（「なぜ 78 万円で試算？」と聞かれる）。

#### 論点2-2. 単位表示（円 or 万円）
- **円単位**: 内部値と一致、桁数が多く（例: `500,000`）入力しづらい。
- **万円単位**: 経営者の思考と整合（「月 50 万円」）、桁が短く入力しやすい。内部で 10,000 倍して保存。

**推奨**: **UI は「万円」単位、内部は円単位で保持**。入力欄の右側に「万円／月」とサフィックス表示。`Inputs.monthlyVendorCost` には `万円値 × 10000` を格納。

#### 論点2-3. バリデーション範囲と刻み
- 最小値: `1`（万円）。0 は商談対象外のため必須入力扱い。
- 最大値: **`10000`（万円 = 1億円/月）** を上限とする。これを超える顧客は当アプリのターゲット外。
- 刻み: `1`（万円単位の整数）。
- 整数制約: 万円単位で整数（内部は円単位で最小 10,000 円刻み）。

### Step 3. 改修費用（単発）の UI を決める

#### 論点3-1. UI 種別
- **推奨**: **数値入力**（月額ベンダー費用と同じ理由）。

#### 論点3-2. 単位
- **推奨**: **万円単位**（月額費用と UI を統一）。サフィックスは「万円／回」と明示。「年 3 回想定」の補足をヘルプテキストで添える（Issue #1 で確定した固定値のコンテキスト提示）。

#### 論点3-3. バリデーション範囲
- 最小値: `0`（万円）。改修費用はゼロがありうる（「改修は発生していない」という回答）。Step 2 と異なり **必須だがゼロ許容**。
- 最大値: **`5000`（万円 = 5,000万円/回）**。
- 刻み: `1`（万円）。

### Step 4. 手作業人数の UI を決める

#### 論点4-1. UI 種別
| 選択肢 | 判断 |
|---|---|
| 数値入力 | 大きな数（例: 50 人）に対応できる。推奨ベース。 |
| ステッパー（± ボタン） | 小さい値（1〜10 人）では直感的。タブレットで操作が速い。 |
| セレクト（1人, 2人, 3人, 5人, 10人, ...） | 商談テンポは速いが粒度の粗さでミスマッチが出る。 |

**推奨**: **ステッパー付き数値入力（hybrid）**。中央に数値入力、両サイドに `−` / `+` ボタン。小規模企業（1〜10 人）はタップで、大企業（50 人等）はキーボード直接入力で対応。

#### 論点4-2. バリデーション範囲
- 最小値: `0`（「手作業は発生していない」の回答を許容）。
- 最大値: **`1000` 人**（これを超える規模は当アプリのターゲット外）。
- **整数制約: 必須**（「0.5 人」は不可）。
- 刻み: `1`。

#### 論点4-3. 単位
- サフィックス「人」。

### Step 5. 更新待ち期間の UI を決める（スピード警告連動）

Issue #1 で **`updateWaitMonths >= 3` で警告発動** が確定済み。UI 種別はこの閾値の見せ方に直結する。

#### 選択肢5-A: セレクト（プリセット）
- 例: `すぐ（1ヶ月未満）` / `1〜2ヶ月` / `3〜6ヶ月` / `6ヶ月以上` / `1年以上`。
- 選択した瞬間に警告の境界が直感的（「3〜6ヶ月」を選ぶと警告）。
- 値の扱い: 各選択肢に代表値（0.5 / 1.5 / 4.5 / 9 / 18 ヶ月 等）を紐付け、`updateWaitMonths` に格納。

#### 選択肢5-B: 数値入力（月単位）
- 自由度は高いが、経営者が「正確な月数」を即答できないケースが多い。
- 例: 「だいたい半年ぐらい」と言われたときに 6 と入れるか 5 と入れるかで警告発動可否が変わる。

#### 選択肢5-C: レンジスライダー（0〜24 ヶ月）
- 直感的だが、0 ヶ月から 3 ヶ月の閾値の近辺で細かい調整が必要。

**推奨**: **選択肢5-A（セレクト / セグメントコントロール）**。理由:
- 経営者が即答できる粒度（「だいたいどのぐらい待たされますか？」）。
- 商談テンポ最速（タップ 1 回）。
- 警告発動の境界（3 ヶ月以上）がラベルと一致するので説明責任クリア。

選択肢の最終案（仕様書で確定）:
| ラベル | 代表値（`updateWaitMonths`） | 警告 |
|---|---|---|
| すぐ対応（〜1ヶ月） | `0.5` | - |
| 1〜2ヶ月 | `1.5` | - |
| 3〜6ヶ月 | `4.5` | **警告発動** |
| 半年〜1年 | `9` | **警告発動** |
| 1年以上 | `18` | **警告発動** |

### Step 6. 内製化状況の UI 種別を最終確定

Issue #1 で **5 段階（0 / 0.25 / 0.5 / 0.75 / 1.0）で確定済み**。仕様書には「セグメントコントロールまたはラジオボタン」と記載。本 Issue で最終確定する。

#### 選択肢6-A: セグメントコントロール（横並び5ボタン）
- タップ 1 回、タブレットで映える、視覚的に状態が一目で伝わる。
- 5 ラベル分の横幅が必要（「完全ベンダー依存」等の日本語ラベルが長い）。

#### 選択肢6-B: ラジオボタン（縦並び5項目）
- どんな画面幅でも確実に収まる。
- タップ領域が若干小さくなりがち。ラベルの視認性は高い。

#### 選択肢6-C: スライダー（離散5ポイントスナップ）
- 「内製化の程度」を直感的な連続量として見せる。
- ただし Issue #1 で離散値を採用済みのため、操作感と離散値のマッピングがズレて違和感を生みやすい。

**推奨**: **選択肢6-A（セグメントコントロール）をタブレット・デスクトップで採用、モバイル縦画面では自動的にラジオ（縦積み）にフォールバック**（レスポンシブ切替）。ラベルを短縮表示するパターン（「完全依存／一部／半分／大半／完全内製」）も併記しておく。

### Step 7. 詳細設定（時給／1日時間／月稼働日数）の UI 詳細

Issue #1 で **折り畳み UI、デフォルト閉じ** が確定済み。本 Issue では折り畳みを開いた後の UI を確定。

#### 論点7-1. 配置とトリガー
- **推奨**: Step-by-Step の最後（確認画面の手前）に「詳細設定を変更する場合はこちら」とアコーディオン。診断ボタンはアコーディオンの下。
  - 理由: 通常商談では触らせず、「根拠を聞かれた時だけ開く」動線が自然。

#### 論点7-2. 各項目の UI 種別と範囲

| 項目 | UI種別 | 単位 | 最小 | 最大 | 刻み | デフォルト | 必須 |
|---|---|---|---|---|---|---|---|
| 時給 | 数値入力 | 円/時 | `500` | `20000` | `100` | `2500` | 任意（未入力でデフォルト適用） |
| 1日作業時間 | 数値入力（小数許容） | 時間/日 | `0.5` | `8` | `0.5` | `2` | 任意 |
| 月稼働日数 | 数値入力 | 日/月 | `1` | `31` | `1`（整数） | `20` | 任意 |

#### 論点7-3. デフォルト値の扱い
- 未入力（空欄）はデフォルト値として扱う（`input.hourlyWage ?? DEFAULT_HOURLY_WAGE`）。
- プレースホルダで「デフォルト: 2,500」を表示し、ユーザーは空欄でもよい。

### Step 8. バリデーション方針を決める

#### 論点8-1. ライブラリ導入
- **選択肢ア**: Zod を導入して `Inputs` スキーマを一元定義。
- **選択肢イ**: 手書き（各フィールドの `onChange` 時に min/max チェック）。

**推奨**: **Zod 導入**。理由:
- `src/lib/calculation.ts` が受け取る `Inputs` 型と UI バリデーションを **単一の真実の源** で管理できる。
- Issue #6 以降のフォーム実装で react-hook-form と組み合わせる際も自然。
- ただし導入時期は Issue #6 以降。本仕様書では「Zod スキーマとして定義する前提」であることと、**スキーマの各フィールドに何を書くか（min/max/integer/optional）** を確定する。

#### 論点8-2. エラー表示タイミング
- **推奨**: 「入力中は控えめに、Blur 時またはステップ送り時にエラー表示」。ステップ送りをブロックする形でバリデーションを効かせる。
- インラインエラー（フィールド直下の赤字 1 行）。文言は「◯ 円以上で入力してください」のように具体的に。

#### 論点8-3. 空値・ゼロ値の扱い
| 項目 | 空値 | ゼロ値 |
|---|---|---|
| 月額ベンダー費用 | NG（必須） | NG（`min: 1`） |
| 改修費用 | NG（必須） | OK |
| 手作業人数 | NG（必須） | OK |
| 更新待ち期間 | NG（セレクト必須選択） | N/A |
| 内製化状況 | NG（ラジオ必須選択） | OK（`0` = 完全ベンダー依存） |
| 詳細設定3項目 | OK（= デフォルト適用） | NG（`min` 未満） |

---

### Step 9. タブレット／スマホ最適化の方針を決める

#### 論点9-1. 数値キーボードの呼び出し
- 金額・人数・月数フィールドは `inputmode="numeric"` を指定、iOS/Android で数値キーボードを即座に開く。
- 小数許容の「1日作業時間」は `inputmode="decimal"`。

#### 論点9-2. タップ領域
- **推奨**: 全インタラクティブ要素は **最小 44 × 44 px**（Apple HIG 推奨）。ステッパーボタン、ラジオ、セレクトは十分な余白を確保。

#### 論点9-3. レスポンシブ
- タブレット縦（768px〜1024px）で最も快適に動くことを第一優先。次点でスマホ縦、デスクトップ。
- タブレット横持ち（商談中の想定姿勢）で 1 ステップが画面に収まりきること。

#### 論点9-4. リアルタイム計算プレビュー
- 入力中に結果の一部（「3年で◯万円」）を画面の端に表示するかは **#3（ダッシュボード）の領分** として切り分け。本仕様書では「リアルタイム再計算するかしないか」の方針だけ記す。

### Zod スキーマの擬似コード（仕様書の中心）
```ts
import { z } from "zod";

const INSOURCING_VALUES = [0, 0.25, 0.5, 0.75, 1.0] as const;
const UPDATE_WAIT_VALUES = [0.5, 1.5, 4.5, 9, 18] as const;

// 注: UI は万円単位で入力。Zod スキーマは「UI 入力値（万円）」に対するバリデーション。
//     `.transform()` で円単位に変換し calculation.ts の Inputs 型に渡す。
export const inputFormSchema = z.object({
  monthlyVendorCostManYen: z.number().int().min(1).max(10_000),
  repairCostManYen:        z.number().int().min(0).max(5_000),
  manualWorkerCount:       z.number().int().min(0).max(1_000),
  updateWaitMonths:        z.number().refine(v => UPDATE_WAIT_VALUES.includes(v as any)),
  insourcingLevel:         z.number().refine(v => INSOURCING_VALUES.includes(v as any)),
  // 詳細設定
  hourlyWage:   z.number().int().min(500).max(20_000).optional(),
  hoursPerDay:  z.number().min(0.5).max(8).optional(),
  daysPerMonth: z.number().int().min(1).max(31).optional(),
});
```

## 4. 考慮事項・リスク

### R1: 万円単位入力と円単位内部値の変換ミス
- UI「50（万円）」を内部 `500_000` 円に変換する際、ゼロ個数の取り違えで 10 倍／1/10 のバグが発生しやすい。
- → 仕様書に「**変換は単一関数 `manYenToYen(v)` / `yenToManYen(v)` で閉じ込める**」と記載し、Issue #6 実装側でそれを徹底する。テストケース（1 万円、100 万円、1 億円）を仕様書付録で例示。

### R2: セレクト型「更新待ち期間」の代表値と警告境界の整合
- 「3〜6ヶ月」カテゴリに代表値 4.5 を割り当てると `>= 3` で警告発動する。境界 `1〜2ヶ月` の代表値 1.5 が `< 3` で警告非発動となる。仕様書に **「代表値はカテゴリの中央値だが、警告閾値との整合を優先して定める」** と明記。

### R3: Zod 導入時期
- Issue #6 で Next.js 雛形を作る際に同時導入するか、Issue #6 後の別 Issue で導入するかの判断が必要。本仕様書では **「Zod 前提で UI 仕様を書き下す」** に留め、導入 Issue 化は Issue #6 のスコープで決める。

### R4: 詳細設定を開いた後に閉じた場合の値保持
- ユーザーが詳細設定を開いて値を変更 → 閉じた場合、入力値は保持するか破棄するか。
- → **保持**（明示的にクリアボタンを押すまで保持）と仕様書に記載。誤タップで閉じても値が消えないことが商談での安心感に直結。

### R5: 0 人（手作業ゼロ）の顧客の扱い
- `manualWorkerCount = 0` だと「創出額＝0」が出る。これは誠実な結果だが営業トークとしては「既に自動化済み、維持フェーズ」に近い。Issue #4（警告コピー）と連携。

### R6: 1項目1ステップの進捗 UI
- 「2/5」等の進捗表示を入れるかは本 Issue で決める。**推奨: 入れる**（完了までのパスが見えていることが経営者のストレス軽減に直結）。

### R7: 戻る・修正操作
- 各ステップで「戻る」ボタン必須、確認画面では各項目に「この項目を修正」リンクを配置。仕様書でワイヤーフレーム相当の記述を入れる。

### R8: 入力値の永続化（リロード耐性）
- 商談中に誤ってブラウザを閉じた場合の復元。
- → **本 Issue のスコープ外**（localStorage 等の永続化は将来拡張）として仕様書の「10. 未解決事項」に記録。

### R9: 「万円」単位表示がグラフ（#3）の表示単位と整合するか
- Issue #1 で結果表示も万円単位と確定済み。**入力単位と結果表示単位が揃う**（万円）ため、ここで違和感は出ない。Issue #3 担当者への明示は必要。
