# Issue #22 実装プラン: 既存 spec ドキュメントの旧パレット参照を新パレットへ置換する

> **本プランのステータス（2026-04-25）**: 草案。§0 の方針 (a)/(b) の最終決定は **PR 着手前に Issue #22 のコメントで合意**を取る運用とする（決定後、本プランの該当節を「確定」へ更新してから着手）。本プランは方針 (a) 推奨を前提に §3 の Before / After を提示するが、(b) 採用時の差分も §0.4 に併記する。

## 0. 採用方針の決定（最優先で読むこと）

### 0.1 背景

本 Issue #22 は、Issue #7（Tailwind デザイントークン）で新パレット「洗練されたニュアンス・キャット」（`canvas` / `ink` / `line` / `accent`）が確定・実装された一方、`docs/spec/*.md` の 3 ファイルに旧パレット（`#F8FAFC` / Blue 500 / Amber 500 / Amber 600 / `slate-900` / `slate-700`）参照が残置されている状態を解消する **文面置換専用 Issue** である。Issue #7 のフォローアップとして起票された経緯（`working/plans/issue-7_tailwind-design-tokens.md` §0 注記）に従う。

### 0.2 最重要意思決定: 旧 Amber 系（警告バナー / 年間創出）の扱い

新パレットはアクセント色が `#9CAEB8`（ミスティ・ブルー、`accent`）の **1 色しかない**。一方、旧パレットの Amber 500 / 600 は以下の **2 系統**で使用されている:

1. **年間の利益創出（メトリクス・グラフ配色）**: 補助カードと積み上げ横棒グラフのうち「3 年間の利益創出」セグメント（`result-dashboard.md` §3.2 / §4.2）。
2. **警告バナー（背景・枠線・アイコン）**: `speedWarning === true` 発動時の `CRITICAL OPPORTUNITY LOSS` バナー（`result-dashboard.md` §8.2 / `warning-copy.md` §3.2 / §6.1 / `pdf-report.md` §5.2）。

これらをどう新パレットに置換するかについて、以下 2 案を比較する。

| 観点 | (a) すべて `accent`（ミスティ・ブルー）に統一 | (b) 警告バナー用に補助色トークンを新規追加（例: くすみオレンジ） |
|---|---|---|
| Tailwind トークン変更 | **不要**（既登録の `accent` のみで運用可） | **必要**（`tailwind.config.ts` への新規 HEX + Tailwind 名追加が前提） |
| 本 Issue のスコープ | 文面置換のみで完結 | コード（`tailwind.config.ts`）にも手を入れる必要があり、Issue 性質が変わる |
| マスター設計書 §3.3 との整合 | ◎（新パレット 4 色のみで完結） | △（マスター設計書に存在しない 5 色目を導入することになり、別途設計上の意思決定が必要） |
| 警告 UI のシグナル強度 | △（背景・枠線・アイコンが accent と同色になるため、CTA・止血カードと色で区別できない。`letterSpacing.warning` + `alert-triangle` アイコン形状 + 全大文字 `CRITICAL OPPORTUNITY LOSS` + 動的金額の組み合わせで補完する設計が必要） | ◎（オレンジ系の専用色で「警告」という別カテゴリを直感に訴求可能） |
| グラフ可読性（積み上げ横棒の止血/3 年創出区別） | △（accent 1 色になるため止血と 3 年創出の区別を別手段で確保する必要あり、§4 で代替案を提示） | ◎（補助色を 3 年創出セグメントに転用可能。ただし「警告 = 利益創出」と色が共有され情報の混線リスク） |
| 進めやすさ | ◎（PR が docs だけになり、レビュー・差分が極小） | △（コード PR が必要。Issue #10 既マージとの整合確認や、命名議論（`warn` / `signal` / `caution` 等）が発生） |
| 後戻りコスト | 低（後で (b) に切り替え可能、トークン追加だけで済む） | 中（一度トークンを増やすと、他の UI でも誘惑的に使われ始め、ブランド統制が緩む懸念） |

### 0.3 推奨: **(a) すべて `accent`（ミスティ・ブルー）に統一**

理由:

1. **本 Issue のスコープ厳守**: イシュー本文が「文面置換」と明示しており、コード変更（Tailwind トークン追加）を持ち込むと PR の性質が混線する。トークン追加が必要なら別 Issue として独立させるのが健全。
2. **マスター設計書 §3.3 への忠実性**: マスター設計書は 4 色構成で確定しており、5 色目を導入すること自体が設計判断としてマスター側の改定を伴う。本 Issue で同時に踏むには重い。
3. **警告シグナル強度は別経路で十分担保可能**: 警告バナーの認知強度は色のみに依存していない。`letterSpacing.warning = 0.06em`（`tailwind.config.ts` で既追加）+ `alert-triangle` アイコン形状 + 大文字英語見出し `CRITICAL OPPORTUNITY LOSS` + 動的月額金額（`warning-copy.md §7.4`）+ `bg-accent/10` 程度の薄い背景 + `border-accent` 1px 枠の組み合わせで、ユーザーは「特別な領域」として認識可能。色相だけが警告 UI の唯一のシグナルではない。
4. **グラフ配色の課題は §4 で別途解決**: accent 1 色化に伴うグラフ区別問題は、`ink` / `accent` の組み合わせまたは `accent` の濃淡で解決できる（§4 参照）。
5. **後戻り可能性**: (a) を採用した後に商談現場のフィードバックで「警告のシグナルが弱い」と判明したら、その時点で別 Issue として補助色トークンを足せばよい（`docs/spec` への再置換は局所的）。一方 (b) を先行採用してしまうと、ブランド統制の観点で 5 色目の存在を後から取り除くのは難しい。

### 0.4 (b) を採用する場合の差分メモ

PR 着手前のコメント協議で (b) が選ばれた場合、本プランの差分は以下:

- §1 の「コード変更なし」は撤回、`tailwind.config.ts` への色追加が本 Issue スコープに含まれる。
- 新規トークン名候補: `signal`（警告シグナル）/ `warn`（伝統的）/ `caution` のいずれか。HEX 候補はマスター設計書の補強提案として `#C7956D`（くすみオレンジ／キャメル）等が考えられるが、本 Issue で導入するならマスター設計書 §3.3 にも併記する追記 PR を別建てで実施する必要あり。
- §3 の Before / After のうち、警告バナー（`result-dashboard.md` §8.2 / `warning-copy.md` §3.2 / §6.1 / §6.2）の置換先は `accent` ではなく新トークン（`signal` 等）になる。
- グラフ配色（§4）も「accent + signal」の 2 色運用が可能になるため、解決案が単純化される（止血 = `accent`、3 年創出 = `signal`）。
- 本 Issue 完了後のフォローアップとして、マスター設計書 §3.3 への 5 色目追記 PR を別途切る必要あり。

### 0.5 PR 着手前の運用

1. 本プランをコミットし、Issue #22 のコメントに「方針 (a) 推奨で進める。1 営業日以内に異論がなければ確定」と投稿。
2. 異論なければ (a) で着手（推奨ルート）。(b) 希望が出た場合は §0.4 の差分を再評価し、トークン追加 PR を本 Issue に含めるか別 Issue 化するかを決定する。
3. 決定後、本プラン §0 ヘッダの「草案」を「確定（採用方針: a / b）」に更新する。

#### 3.1.7 L130（§4.2 色と期間の関係解説）

**Before**:
```
- **色と期間の関係**: Amber は補助カードの「年間の利益創出」とグラフの「3年間の利益創出」で共通して利用する。色は「利益創出」という指標の種類を示し、**期間の別（年間 / 3年）は数値・ラベル側で判別**する。同様に Blue は「止血」指標全般を示し、補助カードとグラフで同色を共用する。
```

**After**:
```

```

## 4. グラフ配色の課題と代替案

### 4.1 課題

旧設計では積み上げ横棒の「止血」「3 年創出」を Blue 500 / Amber 500 という色相の異なる 2 色で区別していた。新パレットには accent（`#9CAEB8`）1 色しかなく、同じやり方では 2 セグメントが視覚的に区別不能になる。色覚多様性対応も含め、別の区別手段を確定する必要がある。

### 4.2 代替案の比較

| 案 | 概要 | メリット | デメリット | 採否 |
|---|---|---|---|---|
| **(I) `accent` の濃淡（不透明度差）** | 止血 = `accent` 100%（`bg-accent`）/ 3 年創出 = `accent` 60%（`bg-accent/60`） | パレット統一感を維持、Tailwind 不透明度ユーティリティで実装可、ブランド一貫性◎ | 色覚多様性ユーザー（特に明度差弱者）には判別しにくい可能性 → 凡例にアイコン併用で補強（§4.3） | **第一推奨** |
| (II) `ink` と `accent` の組み合わせ | 止血 = `ink`（#72665B、ブラウン）/ 3 年創出 = `accent`（#9CAEB8、ミスティ・ブルー） | 色相差が明確で判別容易 | テキスト色（`ink`）をグラフ塗りに使うと「強調 = テキスト」の役割と混線、`canvas` 背景上で `ink` の塗りは想定以上に重く見える可能性 | 次点（第一推奨が現場で問題出たら切替） |
| (III) パターン塗り（縞・ドット） | 止血 = `accent` ベタ / 3 年創出 = `accent` 縞模様 | 色覚に依存しない最強の区別手段 | Recharts でパターン塗りを実装するには SVG パターン定義が必要で実装複雑度が上がる、ブランドトーンの「洗練されたニュアンス」と縞模様が視覚的に衝突する懸念 | 不採用 |
| (IV) `greige` と `accent` の組み合わせ | 止血 = `greige`（#BEB5AA、境界線色）/ 3 年創出 = `accent` | 色相差が出る | `greige` は `line`（境界線）として使われており、塗りに使うと役割が混線。背景 `canvas` (#F8F6F2) との明度差が小さくバーが薄く見える | 不採用 |

### 4.3 第一推奨案 (I) の詳細

- **止血セグメント**: `accent` 不透明度 100%。Recharts の `<Bar fill="#9CAEB8" />` 相当。
- **3 年創出セグメント**: `accent` 不透明度 60%。Recharts では `fill` を `rgba(156, 174, 184, 0.6)` で指定、または fill="#9CAEB8" + `fillOpacity={0.6}`。
- **凡例**: アイコン（`piggy-bank` / `sparkles`）+ ラベル + 金額。アイコン形状で色覚多様性ユーザーへの補強を継続（`result-dashboard.md §4.2` の既存方針を継承）。
- **ツールチップ**: セグメント別に「3 年間の止血: ◯◯ 万円」「3 年間の利益創出: ◯◯ 万円」の文字情報で内訳を補完。
- **凡例の色サンプル**: 凡例内の色マーカーも同じ濃淡（100% / 60%）で表示し、グラフと一致させる。

### 4.4 完全内製顧客（止血 = 0）での挙動

- 止血セグメント（`accent` 100%）が消え、3 年創出（`accent` 60%）のみのバーになる。`accent` の単色で半透明バーが出る形。`result-dashboard.md §4.4` の既存挙動と整合（凡例構造は維持、カード側注記で文脈を語る）。

### 4.5 (b) 採用時の差分

方針 (b) で警告専用色トークンを追加する場合、グラフ配色は次のいずれかを取れる:
- 止血 = `accent`（ミスティ・ブルー）/ 3 年創出 = 新規補助色（くすみオレンジ等）→ Blue/Amber に近い構造を新パレット内で再現可能。
- 補助色は警告とグラフの両方で使うことになるため、「警告 = 利益創出」の意味的混線が生じうる点に注意。マスター設計書 §3.3 への影響評価が必要。

## 8. 考慮事項・リスク

### 8.1 警告バナーの視認性低下リスク

新パレット採用後、警告バナーの色は CTA・補助カード・グラフと同じ `accent`（ミスティ・ブルー）になる。色相のみによる「他要素との差別化」が機能しなくなるため、商談現場で「目立たない」というフィードバックが出るリスクがある。

**緩和策（本 Issue で記述に組み込む）**:
- `letterSpacing.warning = 0.06em`（`tailwind.config.ts` 実装済み）+ 大文字英語見出し + ウェイト 700 で見出しの異質感を確保。
- `alert-triangle` アイコン形状（三角形＋エクスクラメーションマーク）は色相に依らず警告と認識される普遍的形状。
- 動的金額差し込み（`現在、月額 ◯◯ 万円相当の機会損失が発生中`）でインパクトを直接伝達。
- ダッシュボード最上部固定配置（§8.1）で視線優先順位を担保。

**フォローアップ条件**: 上記補強でも視認性不足が判明した場合は、別 Issue で警告専用色トークン追加（方針 (b) 相当）を検討する。本 Issue PR 本文にこの旨を申し送り。

### 8.2 グラフ可読性

`accent` 100% / 60% の濃淡は色覚多様性ユーザー（特に明度差弱者）には判別が難しい可能性がある。凡例アイコン（`piggy-bank` / `sparkles`）+ ツールチップの文字情報併用で補強する設計は維持するが、実装後（Issue #6 以降の本実装 Issue）にユーザーテストで明度差判別性を確認する必要がある。

**フォローアップ条件**: 判別性不足が判明した場合は §4.2 (II) `ink` + `accent` 案または (III) パターン塗り案に切り替える別 Issue を切る。

### 8.3 Issue #4 / Issue #5 の本実装 Issue への影響

本 Issue で `docs/spec/warning-copy.md` / `docs/spec/result-dashboard.md` / `docs/spec/pdf-report.md` の文面が変更されるため、これらを前提とする以下の本実装 Issue が影響を受ける:

- **Issue #4（警告コピー文言の本実装）**: 警告バナーの色定義（背景・枠線・アイコン）が `accent` 系に変わるため、実装時の Tailwind クラスが `bg-amber-500/10` ではなく `bg-accent/10` になる。`warning-copy.md` 改定後の本書を参照する形で実装する。
- **Issue #5（PDF レポート本実装）**: 既に `pdf-report.md` が存在し PDF 仕様は確定しているが、本 Issue で配色記述が更新されるため、Issue #5 の実装段階で `pdf-report.md` の最新版を参照すること。Recharts 経由の `fill` 値が `#9CAEB8` + `fillOpacity` ベースに変わる。
- **Issue #3（結果ダッシュボードの本実装）**: グラフ配色（§4.1 案 (I) 採用）の Recharts 実装方針が確定する。`<Bar fill="#9CAEB8" />` + `<Bar fill="#9CAEB8" fillOpacity={0.6} />` で実装する。

本 Issue の PR 本文で上記 Issue 番号を明示的に参照し、影響範囲を予告する。

### 8.4 マスター設計書との二重管理リスク

マスター設計書 `.claude/ROI診断アプリ「またたび計算機」開発・設計ドキュメント一式.md` §3.3 が既に新パレット正本となっているため、本 Issue の置換後は spec 側がマスター設計書と完全一致する状態になる。今後マスター設計書側で色設計が変更された場合に spec 3 ファイルとの同期管理が必要になるが、これは本 Issue では対応せず、運用上の継続課題として認識する。

### 8.5 「知的色」「猫の温かみ」等の役割解説テキストの取り扱い

旧パレットには色相に紐付いた感覚的形容（「知的色」「猫の温かみ」）が記述されていた。新パレットでも同様の形容を残すかは判断が分かれる:

- **第一推奨**: 形容を削除し、機能的説明に集約する（マスター設計書 §3.3 自体が機能ベースの記述）。
- **代替**: 「ニュアンスのある中性色」「やわらかい質感」等の新パレット向け形容に置換する。レビューで好みが出れば後者へ。

### 8.6 不透明度表記（`bg-accent/10`、`text-ink/80` 等）の Tailwind v3 サポート

Tailwind CSS v3 系は不透明度修飾子（`/10` `/60` `/80` 等）をネイティブサポートしているため、`tailwind.config.ts` への追加変更は不要。spec ドキュメント上の表記もそのまま実装に転記可能。
