## 設計上の考慮点

- **セクション削除 vs 改題**: Issue 本文では「未実装の項目があれば残置、ない場合はセクションごと削除」と提示されているが、削除すると「事前準備 ↔ 本実装」の対応関係が失われ、後で履歴をたどるときに `git log` を頼るしかなくなる。文書の目的（事前準備イシュー実施順プラン）を踏まえると、見出しを「実装完了」に改め、Issue / PR 番号付きの完了ログとして残す方が情報量を維持できる。Issue #69 の受け入れ条件にも違反しない。
- **PR 番号併記の粒度**: 単一 Issue が複数 PR にまたがるケース（#13 → PR #31 + PR #78、#41 → PR #58 + PR #74）は、最初に CLOSE した PR を主、後続を補足として併記する。これにより「最初の完了タイミング」と「最終確定タイミング」が両方追跡できる。
- **派生 Issue の依存表現**: #24 → #50 → #53 → #64 の連鎖は時系列に依存があるため、ガントチャートではフォーク（`├─ / └─`）ではなく `─┬─` で 3 系統に分岐させて表現するか時系列直列で表現するかが分かれる。Issue 本文の「派生」という表現に合わせ、`#24 ─┬─ #50 / #53 / #64` の分岐型を採用した（厳密には #53 は #50 完了後に着手しているが、いずれも #24 から派生したという文脈を優先）。
- **Mermaid 化の見送り**: 既存の依存関係図は Mermaid ではなく ASCII アート（コードブロック）で描かれている。今回の更新でも統一性を保つため ASCII を維持する。Mermaid 化は別 Issue の判断とする。
- **`**現在地**` ブロックの単一情報源化**: 「現在地」と「推奨スケジュール」「ガントチャート」で重複情報が出るため、現在地は要約のみに留め、詳細は推奨スケジュール・ガントチャートに委譲する書き分けを徹底する。
