# 診断結果ダッシュボード仕様書

> 対象: ROI診断アプリ「またたび計算機」 / 関連Issue: #3 / マスター設計書: [`.claude/ROI診断アプリ「またたび計算機」開発・設計ドキュメント一式.md`](../../.claude/ROI診断アプリ「またたび計算機」開発・設計ドキュメント一式.md) §3.2, §3.3 / 前提: [`calculation-logic.md`](./calculation-logic.md), [`input-form.md`](./input-form.md)

本ドキュメントは、マスター設計書 §3.2 の `ResultDashboard` コンポーネント（「アニメーションを用いた数値カウンター、カード型UI」）を実装可能なレベルまで確定するための仕様書です。指標カードの構成・グラフの種別と描画ライブラリ・アニメーション方針・レスポンシブレイアウト・警告バナーの UI 骨格・PDF 連携戦略を本書で確定します。Issue #5（PDF レポート）と Issue #8（依存ライブラリ追加）は本仕様を前提に設計されます。

## 1. 前提と目的

### 1.1 目的
- ダッシュボードで表示する **指標カードの構成とヒエラルキー**を確定する。
- 診断結果を視覚化する **グラフの種類と点数**、および **描画ライブラリ**を確定する。
- 数値カウンター・グラフ描画の **アニメーション方針**を確定する。
- タブレット商談を最優先としたレスポンシブレイアウトと、`speedWarning` 発動時の警告バナー UI 骨格を確定する。
- PDF 出力（Issue #5）との連携方針（同一コンポーネント共用か、スナップショット版を別立てするか）を確定する。
- 表示単位・丸めルールを `calculation-logic.md §6` と完全整合させる徹底方針を確定する。

### 1.2 前提
本仕様書は Issue #1 の成果物 [`calculation-logic.md`](./calculation-logic.md) と Issue #2 の成果物 [`input-form.md`](./input-form.md) に依存する。特に以下の決定事項を前提とする:

- `CalculationResult` 型（`calculation-logic.md §5`）— `threeYearSavings` / `annualProfitCreation` / `threeYearProfitCreation` / `totalThreeYearImpact` / `speedWarning` / `insourcingGap` の 6 プロパティ。
- 内部計算は円単位・浮動小数、UI 表示は **万円単位で四捨五入**（`calculation-logic.md §6`）。
- ダッシュボードの並列表示項目は「3年間の止血」「年間の利益創出」「3年間のトータルインパクト」（`calculation-logic.md §6.1`）。
- PDF 表紙は「3年間のトータルインパクト」を訴求（同上）。
- 最終計算は入力フォームの確認画面「診断する」ボタン押下後に実行（`input-form.md §7.4`）。
- デザイントークン: ベース `#F8FAFC` / アクセント `#3B82F6` (Blue 500) / `#F59E0B` (Amber 500)（マスター設計書 §3.3）。

| 1 | **商談テンポ優先** | ダッシュボード表示後 1〜2 秒でヒーロー数値が頭に入り、30 秒以内に「何が起きているか」を経営者が理解できること。 |

## 3. 情報構造と指標カードの優先順位

### 3.1 ヒーロー数値: **3年間のトータルインパクト**

- `result.totalThreeYearImpact`（＝ `threeYearSavings + threeYearProfitCreation`）をダッシュボード最上段に **最大フォント + 数値カウンター**で表示する。
- フォントサイズは `clamp(2.5rem, 6vw, 4rem)`。中央揃え。
- 数値の直下に小さくフットノート: 「**※ 試算上の最大値**」（`calculation-logic.md §6.1` の規定）。

**採用理由**: `calculation-logic.md §6.1` で PDF 表紙訴求が「3年間のトータルインパクト」と規定済みであり、ダッシュボードと PDF のヒエラルキーを一致させる。合算値をヒーローにすることで「3年でこれだけ価値が動く」という単一メッセージを最初の 1 秒で伝えられる。「3年止血」をヒーローにすると完全内製顧客でヒーローが「0 万円」になり UX が破綻するため不採用。

### 3.2 補助カード構成

ヒーローの直下に、補助カード **2 枚**を横並び（タブレット／デスクトップ）または縦スタック（スマホ）で配置する。

| カード | 表示値 | 副次テキスト | 配色 |
|---|---|---|---|
| **3年間の止血** | `formatManYen(result.threeYearSavings)` | 「ベンダー流出コストの削減」／内製化注記（§3.3） | Blue 500 アクセント |
| **年間の利益創出** | `formatManYen(result.annualProfitCreation)` | 「× 3年 = `formatManYen(result.threeYearProfitCreation)`」の二段表示 | Amber 500 アクセント |

- ヒーロー（トータル）と補助カード（止血／創出）で **同じ式を別表現で語る**構造。`calculation-logic.md §6.1` の「並列表示」要件を守りつつ、視覚的ヒエラルキーを出す。
- `calculation-logic.md §6.1` は「**年間の利益創出**」を明記しているため、カードのメインは年間値、副次に 3 年値を添える二段表示とする。

### 3.3 `insourcingGap` の透明性表示

係数値そのもの（例: `0.75`）をカード化はしない。代わりに止血カードの直下に小さな注記テキストで透明性を担保する。

| 条件 | 注記文言 |
|---|---|
| `insourcingLevel === 0`（完全ベンダー依存） | 注記を表示しない |
| `0 < insourcingLevel < 1` | 「既に内製化されている **◯%** 相当分は削減余地から除外済み」（◯ = `insourcingLevel * 100`） |
| `insourcingLevel === 1`（完全内製） | 止血カード数値を非表示とし、代わりに「**現状、理想形に近い運用のため削減余地は 0 万円**」と誠実表示 |

**採用理由**: 「内製化 75% 除外」とすると「75% 割引された」と誤読されうる。`calculation-logic.md §4.4` のカテゴリ別表示方針（完全内製は「維持フェーズ」として誠実に表示）とトーンを揃える。

### 3.4 レイアウト全体像（ASCII ワイヤー）

```
┌──────────────────────────────────────────────┐
│ [警告バナー (speedWarning=true のときのみ)]     │
├──────────────────────────────────────────────┤
│         ヒーロー: 3年間のトータルインパクト      │
│         ◯◯◯◯ 万円  （カウンター）           │
│         ※ 試算上の最大値                      │
├──────────────────────────────────────────────┤
│  [止血カード]          [年間創出カード]        │
│  ◯◯万円               ◯◯万円               │
│  ※内製化分控除          × 3年 = ◯◯万円       │
├──────────────────────────────────────────────┤
│         積み上げ横棒グラフ (1 本)              │
│         Blue=止血 / Amber=3年創出             │
├──────────────────────────────────────────────┤
│         [PDF ダウンロード] [再診断]           │
└──────────────────────────────────────────────┘
```

> 注記: ワイヤー内の「※内製化分控除」は領域確保のための短縮表記。実際の注記文言は §3.3 に従い「既に内製化されている ◯% 相当分は削減余地から除外済み」を使用する。

## 4. グラフの種類と点数

### 4.1 採用: **積み上げ横棒グラフ 1 本**

- データ: 1 行（ラベル「3年合計」）に **止血（Blue）** と **3年創出（Amber）** を積み上げる。
- グラフの合計バー長がトータルインパクト（ヒーロー数値）と一致する構造で、ヒーローと補助カードとグラフが同じ式を異なる表現で語る。
- 高さ（タブレット）: 120px。スマホ縦: 100px。

**採用理由**:
1. グラフを 1 本に絞ることで商談テンポを保ち、画面占有を抑える。
2. 積み上げ横棒は合算とその内訳を同時に見せられる。
3. 完全内製顧客（`threeYearSavings = 0`）でも止血セグメントが消えて創出のみの単色バーになるだけで、破綻しない。
4. 軸を「万円」で統一でき、`calculation-logic.md §6` と完全整合。
5. 横棒は縦棒と異なり、モバイル縦画面でも値ラベルが被らず堅牢。

**不採用の選択肢**:
- **円グラフ（止血 vs 創出）**: 2 要素円は情報密度が低く、完全内製で全体が創出 100% になる不自然な表示になる。
- **比較バー（現状支出 vs 導入後）**: 「現状支出」の定義が仕様にない。
- **タイムライン（年次推移）**: `threeYearSavings / 3` の按分に根拠がない。
- **ウォーターフォール**: ストーリー性は高いが、Recharts ネイティブに実装がなく、実装難度が 1 本のバーの価値を超える。

### 4.2 配色

| セグメント | 色 | 役割 |
|---|---|---|
| 3年間の止血 | `#3B82F6`（Blue 500 / 知的色） | マスター設計書 §3.3 のアクセント色。 |
| 3年間の利益創出 | `#F59E0B`（Amber 500 / 猫の温かみ） | 同上。 |

- **色と期間の関係**: Amber は補助カードの「年間の利益創出」とグラフの「3年間の利益創出」で共通して利用する。色は「利益創出」という指標の種類を示し、**期間の別（年間 / 3年）は数値・ラベル側で判別**する。同様に Blue は「止血」指標全般を示し、補助カードとグラフで同色を共用する。
- 色覚多様性対応として、凡例ラベルに Lucide アイコン（止血=`piggy-bank`、創出=`sparkles`）を添える。

### 4.3 凡例・ツールチップ

| 要素 | 仕様 |
|---|---|
| 凡例 | グラフ直下に水平配置。アイコン + ラベル + 金額（万円）。 |
| ツールチップ | セグメントをタップ／ホバーで、当該セグメントの金額を `formatManYen` 形式で表示。タブレット主用途のためタップ優先、タップ後 **3 秒で自動クローズ**。 |

### 4.4 完全内製顧客（止血 = 0）での挙動

- Recharts は `threeYearSavings = 0` のセグメントを自然に描画しない（単色バーになる）。
- 凡例は構造を保ち、カード側注記（§3.3）で「現状、理想形に近い運用」を語る方針とする。凡例を動的に除外はしない（視覚構造の安定性を優先）。

## 5. 描画ライブラリの選定

### 5.1 採用: **Recharts**

| 評価軸（重要度順） | Recharts の評価 |
|---|---|
| PDF 出力との相性（Issue #5 が直接依存） | SVG ベース → html2canvas でのラスタライズで劣化最小。◎ |
| 実装速度 | 宣言的 JSX（`<BarChart><Bar /></BarChart>`）で積み上げ横棒が数行で書ける。◎ |
| TypeScript 型サポート | 公式型定義あり。◎ |
| Tailwind / shadcn 周辺親和性 | shadcn/ui 周辺でも採用実績多数。◎ |
| バンドルサイズ | 約 100kB。`next/dynamic` 遅延ロードで初回表示への影響を排除（§5.3 リスク R8）。○ |
| カスタマイズ自由度 | 色・アニメ・ツールチップは props でカスタマイズ可能。猫モチーフ程度なら十分。○ |
| 日本語情報量 | 豊富。○ |

### 5.2 対抗馬の不採用理由

| 候補 | 描画基盤 | 不採用理由 |
|---|---|---|
| Chart.js (+ react-chartjs-2) | Canvas | Canvas を html2canvas でラスタライズすると **二重ラスタライズ**となり、PDF 拡大印刷時の解像度劣化リスクが大きい。 |
| visx (Airbnb) | SVG | 自由度は Recharts を上回るが、本件（積み上げ棒 1 本）には過剰。記述量が増え実装速度が落ちる。 |
| 自前 SVG | SVG | ツールチップ・アニメーション・凡例を自前で作るコストが仕様確定のスピードを毀損する。将来ブランド表現が極端に凝る場合の選択肢として §11 に残す。 |
| D3 直接 | SVG / Canvas | DOM を外側から操作するモデルが React と噛み合わない。Next.js App Router でのマウント制御が煩雑。 |

### 5.3 Issue #8 への申し送り

- 採用バージョン: **Recharts v2 系**（React 18 対応の安定版）。
- 具体的なマイナー版固定は Issue #8（依存ライブラリ追加）のスコープ。`package.json` への追加時にセマンティックバージョニングで `^2.x.y` を検討。
- SSR 境界の注意: `ResultDashboard` / `DashboardView` は **クライアントコンポーネント**として実装する必要あり（`"use client"` 指定、§11 リスク R1）。
- 初回 LCP を守るため、`ResultDashboard` は `next/dynamic` で遅延ロードする方針（§11 リスク R8）。
- 凡例アイコンで使用する `lucide-react` は `.claude/issue-order.md` のフェーズ1 #8 にすでに列挙されているため、本仕様書での追加要求はなし（本仕様書での新規依存は Recharts のみ）。

## 6. アニメーション方針

### 6.1 数値カウンター: **自前 `useCountUp` フック**

| 項目 | 値 |
|---|---|
| 実装方式 | `requestAnimationFrame` ベースの自前フック（50〜100 行） |
| duration | **1,200 ms** |
| イージング | `easeOutCubic` |
| 補間単位 | 円単位で補間し、表示時に `formatManYen` で万円に丸める |

**採用理由**: カウンターは本アプリで唯一の数値アニメーション。Framer Motion / react-spring を入れる理由がこれだけなら割に合わない（バンドル 30〜50kB の追加）。「万円単位に四捨五入した整数」を補間する必要があり、イージングと最終値の丸め整合性（§11 リスク R3）を自前で持った方が安全。

### 6.2 グラフ描画アニメーション: **Recharts 標準**

| 項目 | 値 |
|---|---|
| 実装 | Recharts の `isAnimationActive` / `animationDuration` |
| duration | **800 ms** |
| 発動タイミング | 数値カウンターと**同時開始**（一体感を出す） |

### 6.3 `prefers-reduced-motion` 尊重

| OS 設定 | 挙動 |
|---|---|
| `prefers-reduced-motion: no-preference` | カウンター 1,200ms / グラフ 800ms の標準アニメーション。 |
| `prefers-reduced-motion: reduce` | カウンターは即時最終値表示（`duration: 0`）、グラフも `isAnimationActive={false}`。 |

- メディアクエリ検出は `matchMedia('(prefers-reduced-motion: reduce)')` を `useEffect` 内で一度評価。

### 6.4 発動タイミング

- マウント時に 1 回発動。
- 確認画面から「修正」→ 再度「診断する」で **新しい `result` を受けた場合に再発動**。実装は `result` オブジェクトのハッシュを `key` prop に渡してリマウントを強制する（§11 リスク R7）。

## 8. 警告バナー（Issue #4 連携）UI 骨格

### 8.1 位置

- **ダッシュボード最上部（ヒーローの直上）**。
- 根拠: `speedWarning = true` は「3ヶ月以上の更新待ちで CRITICAL OPPORTUNITY LOSS が発生」という緊急シグナル。数値カードを見る前に認識してもらう必要がある。
- `speedWarning = false` のときはバナーを出さず、ヒーローが最上段に繰り上がる。

### 8.2 色・アイコン

| 要素 | 値 |
|---|---|
| 背景 | Amber 500（#F59E0B）の透明度 10〜15%（`bg-amber-500/10` 相当） |
| 枠線 | Amber 500 実線、1px |
| アイコン | Lucide `alert-triangle`、色 Amber 600（#D97706） |
| テキスト色 | `slate-900` |
| フォントウェイト | 見出し `bold`、本文 `normal` |

### 8.3 余白・高さ

| デバイス | 高さ | 水平パディング | アイコンとテキストの間 |
|---|---|---|---|
| タブレット | 64px（1 行想定） | 24px | 12px |
| スマホ | 72px（2 行改行許容） | 16px | 12px |
| デスクトップ | 64px、最大幅 960px | 24px | 12px |

**文言が長い場合の吸収方針**: 文言は Issue #4 のスコープだが、長さが予測不能のため以下の柔軟性を持たせる:
- バナー高さは **最小 64px・最大 96px** を許容。
- 96px を超える場合は、ヒーロー数値の `font-size` を `clamp` の下限側（`2.5rem`）に切り替えて 1 画面収まりを維持。

### 8.4 アニメーション

- マウント時に **フェードイン 300ms** のみ。ヒーロー数値カウンターと同時に発動してよい。
- 目立たせすぎると他の要素を邪魔するため、スライドインや拡大縮小は行わない。

### 8.5 文言のプレースホルダ化

- 本仕様書では文言は **Issue #4 のスコープ**として扱う。
- Issue #6 の実装では `src/lib/messages.ts`（または `src/lib/warnings.ts`）に **文字列定数 `CRITICAL_OPPORTUNITY_LOSS_MESSAGE`** を置き、`DashboardView` の `warningMessage` props に渡す。Issue #4 完了時はこの定数を差し替えるだけで済む形にする（本仕様書の擬似コードやワイヤー図で `"{{CRITICAL_OPPORTUNITY_LOSS_MESSAGE}}"` と書かれている箇所は、実装ではこの定数参照で置換する）。

## 9. 表示単位と丸めルール

### 9.1 共通フォーマッタ `formatManYen`

`calculation-logic.md §6` の「万円単位で四捨五入、UI に『万円』単位を付与、ゼロ値はそのまま表示」を以下で**単一の真実の源**として実装する。

```ts
/**
 * 円単位の値を「◯◯万円」形式の文字列に変換する。
 * calculation-logic.md §6 準拠: 万円単位で四捨五入、整数、3桁区切り。
 * ゼロ値はハイフン置換せずそのまま "0万円" として返す。
 */
export function formatManYen(yen: number): string {
  const manYen = Math.round(yen / 10_000);
  return `${manYen.toLocaleString("ja-JP")}万円`;
}
```

- Issue #6 実装時に `src/lib/format.ts` として切り出す。
- ゼロ値はそのまま `0万円` と表示（`calculation-logic.md §6` 準拠、ハイフン置換しない）。

### 9.2 桁爆発時の代替表記（大企業顧客対応）

`monthlyVendorCost` 上限 1 億円/月 × 36ヶ月 ＋ 創出で **36 億円級**のヒーロー数値が発生しうる（`input-form.md §4.1` の上限値に基づく）。読みやすさのため、**10 億円以上**のとき `◯億◯万円` 表記に切り替える。閾値を「10 億円」に置くのは、`input-form.md` の上限運用で発生しうる最大値（36 億円級）に対して 1 桁下まで「万円」表記で通し、さらに桁が増えた時点で億表記に切り替える読みやすさの転換点としたため。

```ts
/**
 * ヒーロー数値専用の大桁対応フォーマッタ。
 * 10億円（= 100,000 万円、yen >= 1_000_000_000）以上の場合に「◯億◯万円」表記に切り替える。
 */
export function formatManYenCompact(yen: number): string {
  const manYen = Math.round(yen / 10_000);
  if (manYen < 100_000) {
    return `${manYen.toLocaleString("ja-JP")}万円`;
  }
  const oku = Math.floor(manYen / 10_000);
  const remainder = manYen % 10_000;
  if (remainder === 0) {
    return `${oku.toLocaleString("ja-JP")}億円`;
  }
  return `${oku.toLocaleString("ja-JP")}億${remainder.toLocaleString("ja-JP")}万円`;
}
```

| 適用箇所 | 使い分け |
|---|---|
| ヒーロー数値（トータルインパクト） | `formatManYenCompact` |
| 補助カード数値（止血／創出／3年創出） | `formatManYen`（億単位になる頻度は低い） |
| グラフ軸ラベル／ツールチップ | `formatManYen`（軸の読みやすさ優先） |
| 注記テキスト | `formatManYen` |

### 9.3 表示箇所ごとの適用

| 表示箇所 | フォーマッタ | 補足 |
|---|---|---|
| ヒーローカウンター（完了状態） | `formatManYenCompact` | 補間中は `Math.round(interp / 10_000)` を毎フレーム再計算 |
| 補助カード数値 | `formatManYen` | カウンターあり、同じ補間ロジック |
| グラフ軸ラベル | `formatManYen` | Recharts `tickFormatter` に注入 |
| ツールチップ | `formatManYen` | Recharts `formatter` に注入 |
| 内製化注記（% 表記） | 独自 | `insourcingLevel * 100` を整数 % で表示 |

## 10. PDF スナップショット戦略（Issue #5 連携）

### 10.1 採用: **PDF 専用スナップショット版を別立て**

マスター設計書 §3.2 の `PdfGenerator`（「隠しDOM要素（A4比率）を画像化しPDFに変換」）と完全整合する方針。

### 10.2 責務分離

| レイヤ | 責務 | `animated` prop |
|---|---|---|
| `DashboardView`（描画層） | props ベース、副作用なし。UI の見た目のみを担当 | props で受け取り、カウンター・グラフ両方のアニメーション有無を制御 |
| `ResultDashboard`（画面コンテナ） | `useCountUp` / `prefers-reduced-motion` / レスポンシブ制御 | OS 設定に応じて決定 |
| `PdfDashboard`（PDF 用ラッパ） | 隠し DOM（`position: absolute; left: -9999px;`）、A4 比率固定、`animated={false}` を強制 | 常に `false` |

### 10.3 `DashboardView` 型定義

```ts
interface DashboardViewProps {
  result: CalculationResult;          // calculation-logic.md §5 の型
  insourcingLevel: InsourcingLevel;   // 内製化注記（§3.3）のため
  animated: boolean;                  // 画面: prefers-reduced-motion 次第 / PDF: 常に false
  showWarningBanner: boolean;         // = result.speedWarning
  warningMessage?: string;            // §8.5 の CRITICAL_OPPORTUNITY_LOSS_MESSAGE 定数を渡す
}
```

> 型定義の補足: `animated` と `showWarningBanner` は意図的に **required** とする。`animated` は §11 リスク R10（PDF 版で OS 設定に関わらず `false` を強制する必要）から呼び出し側が必ず明示する設計。`showWarningBanner` も `result.speedWarning` から自明とはいえ、呼び出し側の意図を明確化するため required にする。`warningMessage` のみ Issue #4 連携のためオプショナルとし、未指定時はフォールバック文言もしくは `"{{CRITICAL_OPPORTUNITY_LOSS_MESSAGE}}"` プレースホルダを表示する（§8.5 参照）。

### 10.4 Issue #5 への申し送り

- **html2canvas と SVG フォント**: `PdfDashboard` は system フォントスタックまたは next/font のローカル埋め込みフォントを使う。外部 CSS 依存は作らない（§11 リスク R2）。
- **A4 比率**: A4 縦（210 × 297mm = 595 × 842pt @72dpi）に合わせた隠し DOM を用意。詳細レイアウトは Issue #5 のスコープ。
- **レイアウト流用**: `DashboardView` の props 互換を保ったまま、PDF 側で独自のラッパを作り A4 フィットさせる。画面側の変更で PDF が壊れないことを保証する。

> **確定仕様は [`pdf-report.md`](./pdf-report.md) を参照（Issue #5 で確定）**。A4 内部レイアウト・フォント埋め込み・html2canvas 詳細・`PdfDashboardProps` 型・生成ユーティリティの API はすべて `pdf-report.md §3〜§11` に集約。
