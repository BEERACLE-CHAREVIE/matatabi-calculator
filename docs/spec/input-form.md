## 1. 前提と目的

### 1.1 目的
- 入力フォームの各項目について、**UI 種別・単位・バリデーション範囲・必須／任意** を確定する。
- Step-by-Step UI の区切り方、進捗表示、確認画面の方針を確定する。
- バリデーション方針（Zod 導入前提、エラー表示タイミング）を確定する。
- タブレット／スマホ商談に耐える UX ガイドラインを確定する。

### 1.2 前提
本仕様書は Issue #1 の成果物 [`calculation-logic.md`](./calculation-logic.md) に依存する。特に以下の決定事項を前提とする：

- `Inputs` 型（§5 で定義）
- `InsourcingLevel` の 5 段階（`0 | 0.25 | 0.5 | 0.75 | 1.0`）
- スピード警告の発動条件（`updateWaitMonths >= 3`）
- 詳細設定 3 項目（時給・1日作業時間・月稼働日数）は折り畳み UI でデフォルト閉じ
- UI 表示は万円単位、内部計算は円単位

## 2. 設計原則

| # | 原則 | 意味 |
|---|---|---|
| 1 | **商談テンポ優先** | 1 項目あたり 10 秒以内で入力完了できる UI を目指す。迷うポイントを作らない。 |
| 2 | **経営者が自分で入力できる** | 専門用語・単位換算・キーボード主体の操作は排除。タップで完結することを優先。 |
| 3 | **タブレット最適化第一** | タブレット横持ち（商談姿勢）で 1 ステップが画面に収まりきること。デスクトップは副次。 |
| 4 | **説明責任との両立** | デフォルト値・範囲の根拠は `calculation-logic.md` にリンクバックして答えられる状態にする。 |
| 5 | **誠実な入力設計** | ゼロ値・未入力の挙動は恣意的に盛らず、入力通りの計算結果を返す（Issue #1 §4.4 と整合）。 |

## 3. Step-by-Step UI の区切り方

### 3.1 採用方針: **1 項目 1 ステップ（完全ウィザード方式）**

- 画面に 1 つの質問のみを大きく表示し、「次へ」ボタンで進む。
- 合計 **6 ステップ**（基本 5 項目 + 最終確認画面）。詳細設定は **確認画面（Step 6）の中にアコーディオンとして内包** する（進捗カウントには含めない）。
- 各ステップで **「戻る」ボタンを必須配置**。誤入力時のストレスを最小化する。

### 3.2 進捗インジケータ

- 画面上部に「2 / 5」形式でステップ番号を表示。
- 合わせて横長のプログレスバーを併置し、完了までのパスを視覚化する。
- 詳細設定と確認画面は進捗カウントに含めない（基本 5 項目のみをカウント）。

### 3.3 確認画面

- 全 5 項目の入力値を一覧表示し、各項目の右側に「この項目を修正」リンクを配置（タップで該当ステップへジャンプ）。
- アコーディオン「詳細設定を変更する場合はこちら」を配置。開いた場合、時給・1日時間・月稼働日数の 3 項目を縦並びで表示。
- 画面下部に **「診断する」ボタン**（主要 CTA）。

> PDF レポート内の「入力値サマリー」の表示仕様は [`pdf-report.md §5`](./pdf-report.md) を参照（Issue #5 で確定）。

### 3.4 ステップ構成の全体像

| Step | 内容 | 進捗表示 |
|---|---|---|
| 1 | 月額ベンダー費用 | 1 / 5 |
| 2 | 改修費用 | 2 / 5 |
| 3 | 手作業人数 | 3 / 5 |
| 4 | 更新待ち期間 | 4 / 5 |
| 5 | 内製化状況 | 5 / 5 |
| 6 | 確認画面（+ 詳細設定アコーディオン） | 「5 / 5」表示継続、プログレスバーは 100% で固定 |

## 4. 基本 5 項目の UI 仕様

### 4.1 月額ベンダー費用 (`monthlyVendorCost`)

| 項目 | 値 |
|---|---|
| UI 種別 | **数値入力**（テキストフィールド） |
| 単位 | **万円／月**（サフィックス表示） |
| 内部保持 | 円単位（`UI 入力値 × 10_000`） |
| 最小値 | `1`（万円） |
| 最大値 | `10_000`（万円 = 1 億円／月） |
| 刻み | `1`（整数） |
| 必須／任意 | **必須**（ゼロ不可） |
| `inputmode` | `numeric` |
| プレースホルダ | `例: 50` |
| ヘルプ文 | 「現在ベンダーに支払っている IT 関連の月額費用（保守・運用・開発受託等の合計）」 |

**採用理由**: 金額の正確性が必要（スライダーでは説明責任が揺らぐ）。万円単位にすることで桁数が抑えられ、経営者の「月 50 万円」という思考と整合する。

### 4.2 改修費用 (`repairCost`)

| 項目 | 値 |
|---|---|
| UI 種別 | **数値入力** |
| 単位 | **万円／回**（サフィックス表示） |
| 内部保持 | 円単位（`UI 入力値 × 10_000`） |
| 最小値 | `0`（万円、ゼロ許容） |
| 最大値 | `5_000`（万円 = 5,000 万円／回） |
| 刻み | `1`（整数） |
| 必須／任意 | **必須**（ゼロ値は許容） |
| `inputmode` | `numeric` |
| プレースホルダ | `例: 30` |
| ヘルプ文 | 「1 回あたりの改修・機能追加の発注費用。年 3 回想定で試算します（`calculation-logic.md` §2）」 |

**採用理由**: 「改修は発生していない」という回答（= 0）を許容。月額費用と UI を統一。年 3 回想定の補足を明示して説明責任を果たす。

### 4.3 手作業人数 (`manualWorkerCount`)

| 項目 | 値 |
|---|---|
| UI 種別 | **ステッパー付き数値入力**（数値フィールド両サイドに `−` / `+` ボタン） |
| 単位 | **人**（サフィックス表示） |
| 最小値 | `0`（手作業なしを許容） |
| 最大値 | `1_000`（人） |
| 刻み | `1` |
| 整数制約 | **必須**（小数不可） |
| 必須／任意 | **必須**（ゼロ値は許容） |
| `inputmode` | `numeric` |
| プレースホルダ | `例: 5` |
| ヘルプ文 | 「手作業・定型業務に従事している人数。AI 自動化の対象人数を試算します」 |

**採用理由**: 小規模企業（1〜10 人）はステッパーで直感的に、大企業（50 人以上）はキーボード入力で対応できるハイブリッド設計。0 人は「既に自動化済み／維持フェーズ」の誠実な表示に繋がる。

### 4.4 更新待ち期間 (`updateWaitMonths`)

Issue #1 で **`updateWaitMonths >= 3` で警告発動**が確定済み。UI はこの閾値の直感的な見せ方を優先する。

| 項目 | 値 |
|---|---|
| UI 種別 | **セレクト（セグメントコントロール / ラジオ）**、離散 5 択 |
| 内部値の単位 | ヶ月（UI 上は選択肢ラベルに内包され、単位記号は表示されない） |
| 必須／任意 | **必須**（未選択は次ステップへ進めない） |
| デフォルト選択 | なし（全ボタンをニュートラル表示で提示し、明示選択を促す） |

#### 選択肢と代表値

| ラベル | 代表値 (`updateWaitMonths`) | スピード警告 |
|---|---|---|
| すぐ対応（〜1ヶ月） | `0.5` | — |
| 1〜2ヶ月 | `1.5` | — |
| 3〜6ヶ月 | `4.5` | **発動** |
| 半年〜1年 | `9` | **発動** |
| 1年以上 | `18` | **発動** |

**採用理由**: 経営者が即答できる粒度（「だいたいどのぐらい？」）。タップ 1 回で完了し商談テンポが最速。警告境界（3 ヶ月以上）がラベル境界と一致して説明責任を満たす。

> 代表値は原則カテゴリの中央値だが、**警告閾値との整合を最優先**に設定している（例: 「3〜6ヶ月」を 4.5 にすることで `>=3` のトリガを確実に発動）。

### 4.5 内製化状況 (`insourcingLevel`)

Issue #1 で 5 段階が確定済み。本仕様書で UI 種別を最終確定する。

| 項目 | 値 |
|---|---|
| UI 種別 | **セグメントコントロール（レスポンシブでラジオにフォールバック）** |
| 必須／任意 | **必須**（未選択は次ステップへ進めない） |
| デフォルト選択 | なし（全ボタンをニュートラル表示で提示し、明示選択を促す） |

#### 選択肢

| フルラベル | 短縮ラベル | 内部値 (`insourcingLevel`) | 係数 (`insourcingGap`) |
|---|---|---|---|
| 完全ベンダー依存 | 完全依存 | `0` | `1.0` |
| 一部内製 | 一部 | `0.25` | `0.75` |
| 半分内製 | 半分 | `0.5` | `0.5` |
| 大半内製 | 大半 | `0.75` | `0.25` |
| 完全内製 | 完全 | `1.0` | `0.0` |

#### レスポンシブ切替

- **タブレット・デスクトップ（幅 ≥ 640px）**: 横並び 5 ボタンのセグメントコントロール。ラベルは短縮形を採用し横幅を抑える。ホバー／タップで補足にフルラベルを表示。
- **モバイル縦画面（幅 < 640px）**: 縦積みラジオボタンにフォールバック。ラベルはフルラベルを使用。

**採用理由**: 商談中はタブレットが主、いずれの画面幅でも確実に表示されること、5 段階の視覚的一覧性を担保すること。

## 5. 詳細設定 3 項目の UI 仕様（折り畳み内）

### 5.1 配置とトリガー

- **Step 6（確認画面）内に配置**。「詳細設定を変更する場合はこちら」というアコーディオンヘッダでデフォルトは閉じた状態。
- アコーディオンの下に「診断する」ボタン（主要 CTA）。通常商談では触らせず、根拠を問われた時のみ開く動線。
- アコーディオンを開いて値を変更 → 閉じた場合でも、**入力値は保持**する（明示的なクリアボタン操作まで）。

### 5.2 各項目の仕様

| 項目 | UI 種別 | 単位 | 最小 | 最大 | 刻み | デフォルト | 必須 | `inputmode` |
|---|---|---|---|---|---|---|---|---|
| 時給 (`hourlyWage`) | 数値入力 | 円／時 | `500` | `20_000` | `100` | `2_500` | 任意 | `numeric` |
| 1日作業時間 (`hoursPerDay`) | 数値入力（小数可） | 時間／日 | `0.5` | `8` | `0.5` | `2` | 任意 | `decimal` |
| 月稼働日数 (`daysPerMonth`) | 数値入力 | 日／月 | `1` | `31` | `1`（整数） | `20` | 任意 | `numeric` |

### 5.3 未入力時の挙動

- 空欄はデフォルト値として扱う（`input.hourlyWage ?? DEFAULT_HOURLY_WAGE`）。
- プレースホルダで「デフォルト: 2,500」のように表示し、空欄のまま進められることを明示する。
- 空欄はエラーにしない。**値が入力された場合のみ min/max バリデーションを適用**。

### 5.4 ヘルプ文

アコーディオン冒頭に以下の注記を表示：

> これらの値は業界標準値で試算しています。実態に合わせて変更したい場合は個別に入力してください。

## 6. バリデーション方針

### 6.1 ライブラリ: **Zod を採用**

- `src/lib/calculation.ts` が受け取る `Inputs` 型と UI バリデーションを **単一の真実の源** で管理できる。
- Issue #6 以降で react-hook-form と組み合わせる想定。
- Zod の実際の導入（`package.json` への追加）は Issue #6 以降のスコープ。本仕様書では「Zod 前提の UI スキーマ」を擬似コードで確定する。

### 6.2 エラー表示タイミング

| タイミング | 挙動 |
|---|---|
| 入力中 (`onChange`) | **エラーは表示しない**（商談中に赤字が連続点灯するのを避ける） |
| フィールド離脱 (`onBlur`) | インラインエラー表示（フィールド直下、1 行赤字） |
| ステップ送り (`next` クリック) | バリデーション失敗時は **ステップ送りをブロック**し、該当フィールドにフォーカス＋エラー表示 |

### 6.3 エラー文言のトーン

- 敬体（「〜してください」）。
- 具体値を含める（例: 「1 万円以上で入力してください」）。
- 技術的表現は避ける（例: 「数値のみ」ではなく「半角数字で入力してください」）。

### 6.4 空値・ゼロ値の扱い

空値と範囲外（ゼロを含む）はバリデーション上の根拠が異なるため列を分けて記載する。

| 項目 | 空値 | 空値の根拠 | ゼロ値 | ゼロ値の根拠 |
|---|---|---|---|---|
| 月額ベンダー費用 | NG | `required`（必須） | NG | `.min(1)` |
| 改修費用 | NG | `required`（必須） | OK | `.min(0)` が許容 |
| 手作業人数 | NG | `required`（必須） | OK | `.min(0)` が許容 |
| 更新待ち期間 | NG | セレクト未選択（`required`） | N/A | 選択肢にゼロ値無し |
| 内製化状況 | NG | セグメント未選択（`required`） | OK | `0` = 完全ベンダー依存を選択済み扱い |
| 詳細設定 3 項目 | OK | 空欄 = デフォルト適用（`.optional()`） | NG | 各 `min` 未満のため（例: `hourlyWage.min(500)`） |

### 6.5 Zod スキーマ擬似コード

> 本擬似コードが Issue #6 以降の実装で参照される正本です。

#### 刻み（step）の検証方針

- **§4・§5 の表に定義した「刻み」は HTML `<input step="...">` で UI 層に担保させ、Zod では範囲のみを検証する** を基本方針とする。
  - 理由: `inputmode="numeric"` や `<input type="number">` のスピナー操作経由では刻み外の値が入り込まないため、Zod 側で再検証する必要性が低い。
  - 例外: 「手作業人数」「月稼働日数」の **整数制約**（`.int()`）と、金額系の **整数制約**（`.int()`）は Zod で強制する（貼り付け等で小数値が入り込むケースを弾くため）。
- 時給の刻み `100`、1日作業時間の刻み `0.5` は Zod で `.multipleOf()` を使わず、UI の `step` 属性のみで担保する（実装者は `<input step="100">` / `<input step="0.5">` を指定すればよい）。

```ts
import { z } from "zod";

// 内製化状況の許容値（calculation-logic.md §4.1 と整合）
const INSOURCING_VALUES = [0, 0.25, 0.5, 0.75, 1.0] as const;

// 更新待ち期間の代表値（§4.4 と整合）
const UPDATE_WAIT_VALUES = [0.5, 1.5, 4.5, 9, 18] as const;

/**
 * UI 入力値に対するスキーマ。
 * 金額系は「万円単位」で受け取り、計算レイヤーへ渡す前に円単位に変換する（§9 参照）。
 * 刻みは UI 層（HTML `step` 属性）で担保し、ここでは範囲と整数制約のみを検証する。
 */
export const inputFormSchema = z.object({
  monthlyVendorCostManYen: z
    .number({ invalid_type_error: "半角数字で入力してください" })
    .int("整数で入力してください")
    .min(1, "1 万円以上で入力してください")
    .max(10_000, "10,000 万円以下で入力してください"),

  repairCostManYen: z
    .number({ invalid_type_error: "半角数字で入力してください" })
    .int("整数で入力してください")
    .min(0, "0 以上で入力してください")
    .max(5_000, "5,000 万円以下で入力してください"),

  manualWorkerCount: z
    .number({ invalid_type_error: "半角数字で入力してください" })
    .int("整数で入力してください")
    .min(0, "0 以上で入力してください")
    .max(1_000, "1,000 人以下で入力してください"),

  updateWaitMonths: z
    .number()
    .refine((v) => (UPDATE_WAIT_VALUES as readonly number[]).includes(v), {
      message: "選択肢から 1 つ選んでください",
    }),

  insourcingLevel: z
    .number()
    .refine((v) => (INSOURCING_VALUES as readonly number[]).includes(v), {
      message: "選択肢から 1 つ選んでください",
    }),

  // 詳細設定（任意。未入力時はデフォルト値を適用）
  hourlyWage: z
    .number()
    .int("整数で入力してください") // 時給は整数円。刻み 100 は UI の step 属性で担保
    .min(500, "500 円以上で入力してください")
    .max(20_000, "20,000 円以下で入力してください")
    .optional(),

  hoursPerDay: z
    .number() // 小数許容（0.5 時間単位）。刻み 0.5 は UI の step 属性で担保
    .min(0.5, "0.5 時間以上で入力してください")
    .max(8, "8 時間以下で入力してください")
    .optional(),

  daysPerMonth: z
    .number()
    .int("整数で入力してください")
    .min(1, "1 日以上で入力してください")
    .max(31, "31 日以下で入力してください")
    .optional(),
});

export type InputFormValues = z.infer<typeof inputFormSchema>;
```

## 7. タブレット／スマホ最適化ガイドライン

### 7.1 数値キーボードの呼び出し

- 金額・人数・月数フィールドは `inputmode="numeric"` を指定し、iOS／Android で即座に数値キーボードが開くようにする。
- 小数許容の「1日作業時間」は `inputmode="decimal"` を指定。

### 7.2 タップ領域

- 全インタラクティブ要素（ボタン・ラジオ・セグメント・ステッパー `−` / `+`）は **最小 44 × 44 px**（Apple HIG 準拠）を確保する。
- 隣接する要素間に最低 8 px のマージンを設け、誤タップを防ぐ。

### 7.3 レスポンシブブレークポイント

| 優先度 | 画面幅 | 想定デバイス |
|---|---|---|
| 1（最優先） | 768px 〜 1024px | タブレット縦／横（商談姿勢） |
| 2 | 〜 640px | スマホ縦 |
| 3 | 1024px 〜 | デスクトップ |

- **タブレット横持ちで 1 ステップが画面に収まりきる**ことを必須要件とする（スクロール不要）。
- 内製化状況は 640px 境界でセグメント → ラジオにフォールバック（§4.5）。

### 7.4 リアルタイム計算プレビュー

## 8. エラー表示・プレースホルダ戦略

### 8.1 プレースホルダ

- 全数値入力フィールドで `例: <代表値>` 形式を使用（例: `例: 50`）。
- 詳細設定 3 項目はプレースホルダに `デフォルト: 2,500` のようにデフォルト値を明示し、空欄のまま進められることを示す。

### 8.2 エラー表示位置

- フィールド直下に 1 行（赤文字 + アイコン）。
- セレクト／ラジオはグループ全体の下に表示。

### 8.3 エラー文言一覧（代表例）

| 項目 | 条件 | 文言 |
|---|---|---|
| 月額ベンダー費用 | 空欄 | 月額費用を入力してください |
| 月額ベンダー費用 | `< 1` | 1 万円以上で入力してください |
| 月額ベンダー費用 | `> 10_000` | 10,000 万円以下で入力してください |
| 改修費用 | 空欄 | 改修費用を入力してください（発生していない場合は 0） |
| 改修費用 | `< 0` | 0 以上で入力してください |
| 手作業人数 | 空欄 | 手作業人数を入力してください |
| 手作業人数 | 小数 | 整数で入力してください |
| 更新待ち期間 | 未選択 | 選択肢から 1 つ選んでください |
| 内製化状況 | 未選択 | 選択肢から 1 つ選んでください |
| 時給（詳細） | `< 500` | 500 円以上で入力してください |

#### 8.3.1 `inputmode` だけでは防げないケースへの対応方針

`inputmode="numeric"` / `inputmode="decimal"` はソフトウェアキーボードの種類を指定するだけで、**ペースト操作・物理キーボード接続・OS によっては負号や小数の入力を物理的には防げない**。そのため以下の方針を取る。

| ケース | 対応層 | 方針 |
|---|---|---|
| 負値のペースト（例: `-100` を貼り付け） | **Zod（`.min(0)` / `.min(1)`）** | 入力は受け付けるが、バリデーション時に弾き §8.3 のエラー文言を表示 |
| 小数のペースト（例: 手作業人数に `1.5` を貼り付け） | **Zod（`.int()`）** | 同上。整数制約に反するためエラー |
| 全角数字のペースト（例: `１００`） | UI 層で正規化推奨 | `onChange` 時に `Number()` で変換を試行、失敗時は `invalid_type_error` でエラー |

**原則**: UI の `inputmode` / `type="number"` は **UX 改善のため**（適切なキーボードを出す）に使い、**入力検証は Zod で必ず行う**。UI のみを頼らない。

## 9. フォームと計算ロジックの単位変換ルール

### 9.1 原則

- **UI 入力単位は万円、内部計算単位は円**。両者の変換は **単一の変換関数に閉じ込める**。
- 変換ミス（ゼロ個数の取り違え）は致命的なバグに直結するため、Issue #6 の実装では以下の命名で関数を用意する：

```ts
export const manYenToYen = (manYen: number): number => manYen * 10_000;
export const yenToManYen = (yen: number): number => yen / 10_000;
```

### 9.2 `InputFormValues` → `Inputs` への変換

```ts
// UI レイヤの値（万円単位）を calculation.ts の Inputs（円単位）に変換する
function toCalculationInputs(form: InputFormValues): Inputs {
  return {
    monthlyVendorCost: manYenToYen(form.monthlyVendorCostManYen),
    repairCost:        manYenToYen(form.repairCostManYen),
    manualWorkerCount: form.manualWorkerCount,
    updateWaitMonths:  form.updateWaitMonths,
    insourcingLevel:   form.insourcingLevel as InsourcingLevel,
    hourlyWage:        form.hourlyWage,
    hoursPerDay:       form.hoursPerDay,
    daysPerMonth:      form.daysPerMonth,
  };
}
```

### 9.3 変換テストケース（Issue #6 実装時の必須テスト）

| UI 入力値 | 期待される内部値（円） |
|---|---|
| `1`（万円） | `10_000` |
| `100`（万円） | `1_000_000` |
| `10_000`（万円、上限） | `100_000_000`（1 億円） |
| `0`（万円、改修費用の下限） | `0` |
