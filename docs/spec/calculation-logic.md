# 計算ロジック仕様書

> 対象: ROI診断アプリ「またたび計算機」 / 関連Issue: [#1](https://github.com/) / マスター設計書: [`.claude/ROI診断アプリ「またたび計算機」開発・設計ドキュメント一式.md`](../../.claude/ROI診断アプリ「またたび計算機」開発・設計ドキュメント一式.md) §3.1

本ドキュメントは、マスター設計書 §3.1「計算ロジック詳細」を実装可能なレベルまで確定するための仕様書です。Issue #2〜#5（入力フォーム／グラフ／警告コピー／PDF）は本仕様を前提に設計されます。

---

## 1. 前提と目的

### 1.1 目的
- ROI 計算式で使用される固定値の扱い（固定 / デフォルト値 / 可変）を確定する。
- 「内製化状況」入力の選択肢と、計算式への反映方法を確定する。
- 最終計算式を擬似コードレベルで明文化し、後続実装（Issue #6以降）が参照できる状態にする。

### 1.2 設計原則
- **商談テンポ優先**: 通常フォームは 5 項目に留め、タブレット商談で快適に入力できること。
- **説明責任**: 経営者に数値の根拠を問われた時、営業担当が即答できる前提値設計であること。
- **誠実な結果**: 既に理想形に近い顧客（例: 完全内製）には、誠実に「削減余地なし」と表示できること。

## 2. 固定値一覧と扱い

| 変数 | 値 | 扱い | 理由 |
|---|---|---|---|
| 試算期間 (`MONTHS_IN_PERIOD`) | 36 ヶ月 | **固定** | 商品コンセプト「3年間の止血」に結合。可変化は不可。 |
| 試算年数 (`YEARS_IN_PERIOD`) | 3 年 | **固定** | 同上。 |
| 年あたり改修回数 (`REPAIRS_PER_YEAR`) | 3 回 | **固定** | マスター設計書準拠。「四半期ごとを想定し軽微な1回を除外」の解釈で根拠付け。 |
| 月あたり月数 (`MONTHS_PER_YEAR`) | 12 ヶ月 | **固定** | 暦定数。 |
| 時給単価 (`hourlyWage`) | 2,500 円/時 | **デフォルト値（可変）** | 顧客業種によって実質人件費は大きく異なるため、詳細設定で上書き可とする。 |
| 1日あたり作業時間 (`hoursPerDay`) | 2 時間/日 | **デフォルト値（可変）** | 顧客の定型業務の実態により変動するため。 |
| 月あたり稼働日数 (`daysPerMonth`) | 20 日/月 | **デフォルト値（可変）** | 業種により稼働日数が異なるため。 |

### 2.1 UI への影響
- 通常フォームは 5 項目のまま（#2 の設計前提）。
- 「時給 / 1日作業時間 / 月稼働日数」の 3 項目は **「詳細設定」折り畳み UI** として配置し、デフォルトでは閉じた状態。
- 折り畳みを開かない限り、デフォルト値が使用される。

## 3. デフォルト値の根拠（説明責任用）

商談で「なぜこの値なのか」を問われた時の回答ロジック。厳密な統計出典ではなく、営業現場で即答できるレベルの説明を担保する。

| 値 | 説明ロジック（1行） |
|---|---|
| 時給 2,500 円 | 中小企業の平均的なホワイトカラー人件費（社会保険料等込みの実質コスト）を参照した標準値。 |
| 1日 2 時間 | 定型業務・手作業の 1 人あたり平均所要時間として業界で一般的に使われる目安値。 |
| 月 20 日 | 標準的な営業日数（週5日 × 4週）。 |
| 年 3 回改修 | Web/システムの典型的な改修頻度（四半期ごとを想定し、そのうち 1 回は軽微対応として除外）。 |
| 36 ヶ月（3年） | 商品コンセプト「3年間の投資対効果」に準拠。中期経営計画の標準期間。 |

> 注記: 各値は仮の説明ロジックであり、将来的に統計出典に置き換える余地を残す。UI には「業界標準値で試算」と記載し、詳細設定で上書き可能であることを明示する。

## 4. 「内製化状況」の選択肢と係数定義

### 4.1 選択肢
5 段階のセグメントコントロール（またはラジオボタン）で提示する。

| ラベル | 内部値 (`insourcingLevel`) | 係数 (`insourcingGap`) |
|---|---|---|
| 完全ベンダー依存 | `0` | `1.0` |
| 一部内製 | `0.25` | `0.75` |
| 半分内製 | `0.5` | `0.5` |
| 大半内製 | `0.75` | `0.25` |
| 完全内製 | `1.0` | `0.0` |

### 4.2 計算式への反映方法
- **係数として「3年間の止血」に乗算する**（方式 ii）。
- 計算式: `insourcingGap = 1 - insourcingLevel`
- 説明ロジック: 「既に内製化されている割合分は既に自社内でコストが吸収済みなので、ベンダー流出コストからは除外」。

### 4.3 「創出」には乗算しない理由
- AI 自動化による手作業削減（創出）は、ベンダー委託の有無と独立した効果。
- 完全内製の企業でも手作業は発生しうるため、創出額は内製化状況に影響されない。

### 4.4 カテゴリ別表示文の切替（併用）
計算式への係数化に加え、カテゴリごとに訴求文を出し分ける（Issue #4 で詳細文言を確定）。

| カテゴリ | 表示方針 |
|---|---|
| 完全ベンダー依存 / 一部内製 | 「削減余地大」の訴求 |
| 半分内製 / 大半内製 | 「更なる効率化の提案」訴求 |
| 完全内製 | 「既に理想形、維持フェーズ」として誠実に表示 |

## 5. 最終計算式（擬似コード）

本擬似コードが Issue #6 以降の実装で参照される正本です。

```ts
// ───────────────── 定数（固定） ─────────────────
const MONTHS_IN_PERIOD = 36;        // 試算期間（3年）
const YEARS_IN_PERIOD = 3;
const REPAIRS_PER_YEAR = 3;          // 年あたり改修回数
const MONTHS_PER_YEAR = 12;

// ───────────────── デフォルト値（ハイブリッドで上書き可） ─────────────────
const DEFAULT_HOURLY_WAGE = 2500;    // 円/時
const DEFAULT_HOURS_PER_DAY = 2;     // 時間/日
const DEFAULT_DAYS_PER_MONTH = 20;   // 日/月

// ───────────────── ユーザー入力 ─────────────────
type InsourcingLevel = 0 | 0.25 | 0.5 | 0.75 | 1.0;

interface Inputs {
  // 基本5項目
  monthlyVendorCost: number;           // 月額ベンダー費用（円）
  repairCost: number;                  // 単発改修費用（円）
  manualWorkerCount: number;           // 手作業人数
  updateWaitMonths: number;            // 更新待ち期間（月）
  insourcingLevel: InsourcingLevel;    // 内製化状況

  // 詳細設定（オプション。未指定時はデフォルト値）
  hourlyWage?: number;
  hoursPerDay?: number;
  daysPerMonth?: number;
}

interface CalculationResult {
  threeYearSavings: number;            // 3年間の止血（削減額）
  annualProfitCreation: number;        // 年間の利益創出額
  threeYearProfitCreation: number;     // 3年間の利益創出額
  totalThreeYearImpact: number;        // 3年間のトータルインパクト（合算）
  speedWarning: boolean;               // スピード警告発動フラグ
  insourcingGap: number;               // 内製化ギャップ係数（0〜1）
}

function calculate(input: Inputs): CalculationResult {
  const hourlyWage = input.hourlyWage ?? DEFAULT_HOURLY_WAGE;
  const hoursPerDay = input.hoursPerDay ?? DEFAULT_HOURS_PER_DAY;
  const daysPerMonth = input.daysPerMonth ?? DEFAULT_DAYS_PER_MONTH;

  // 内製化ギャップ係数（既に内製化された分は削減余地から除外）
  const insourcingGap = 1 - input.insourcingLevel;

  // 3年間の止血（削減額）: 内製化ギャップを乗算
  const rawSavings =
    input.monthlyVendorCost * MONTHS_IN_PERIOD +
    input.repairCost * REPAIRS_PER_YEAR * YEARS_IN_PERIOD;
  const threeYearSavings = rawSavings * insourcingGap;

  // 年間の利益創出額: 内製化係数は乗算しない
  const annualProfitCreation =
    input.manualWorkerCount *
    hoursPerDay *
    daysPerMonth *
    MONTHS_PER_YEAR *
    hourlyWage;

  const threeYearProfitCreation = annualProfitCreation * YEARS_IN_PERIOD;

  return {
    threeYearSavings,
    annualProfitCreation,
    threeYearProfitCreation,
    totalThreeYearImpact: threeYearSavings + threeYearProfitCreation,
    speedWarning: input.updateWaitMonths >= 3,
    insourcingGap,
  };
}
```

## 6. 表示・丸めルール

| 項目 | ルール |
|---|---|
| 内部計算 | 円単位・浮動小数で保持（精度を落とさない） |
| UI 表示単位 | **万円単位** で丸めて表示（経営者向けに可読性優先） |
| 丸め方 | **四捨五入**（`Math.round(value / 10000)` の形） |
| ゼロ値 | `0` 円でもそのまま表示（ハイフン等に置換しない） |
| 負値 | 計算上は発生しない前提。発生時は `0` として表示し、バリデーション強化を検討 |

### 6.1 表示項目
- ダッシュボード: 「3年間の止血」「年間の利益創出」「3年間のトータルインパクト」を並列表示
- PDF サマリー: 3年間のトータルインパクトを表紙訴求
- 注記: UI には「試算上の最大値」である旨を明記する

## 7. スピード警告の発動条件

- 条件: `updateWaitMonths >= 3`（3ヶ月以上で発動）
- 表示: マスター設計書通り「CRITICAL OPPORTUNITY LOSS」を含む警告 UI（文言の最終版は Issue #4 で確定）

## 8. 未解決事項 / 将来拡張

本 Issue のスコープ外として記録し、必要に応じて別 Issue 化する。

- **業種別プリセット**: 製造業・IT業・小売業等で時給/稼働日数のプリセットを持つ拡張。
- **統計出典の置き換え**: 現在の説明ロジックを、公的統計データ（賃金構造基本統計調査等）で裏付ける作業。
- **多通貨対応**: 現時点では円のみ。
- **年度を跨ぐ価格改定への対応**: ベンダー費用が年度で変動する場合の取り扱い。
